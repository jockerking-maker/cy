## 一、BlueStore 深度原理

### 1.1 为什么 BlueStore 取代了 FileStore？

**FileStore 的痛点：**
```
对象写入 → 交给本地文件系统（XFS）
    ↓
文件系统写 Journal → 写数据
    ↓
Ceph 自己还要写一遍 WAL
```
❌ 问题：两层日志，两次刷盘，性能差；还要受文件系统的限制。

**BlueStore 的设计：**
```
对象写入 → BlueStore 直接管理裸盘
    ↓
写 RocksDB（元数据） → 写数据到裸盘
```
✅ 好处：
- 跳过文件系统，直接管磁盘，少了一层开销
- 元数据存在 RocksDB，查询更快
- 支持内联校验和，数据更可靠
- 总体性能比 FileStore 高 30%~50%

> 💡 FileStore 已经是过去式了，新版本 Ceph 全是 BlueStore，不用研究 FileStore 了。

### 1.2 BlueStore 三大分区

```
┌───────────────────────────────────────────────────────┐
│                    整块磁盘                            │
├──────────┬──────────────┬────────────────────────────┤
│   WAL    │      DB      │          Block             │
│ (1-2G)   │   (10-30G)   │       (剩下所有空间)        │
└──────────┴──────────────┴────────────────────────────┘
```

| 分区 | 存什么 | 推荐介质 | 大小 |
|------|--------|---------|------|
| **WAL** | 写入日志，先写 WAL 再写数据 | 越快越好，NVMe 最佳 | 每个 OSD 1~2G |
| **DB** | RocksDB 数据，对象元数据、omap | SSD/NVMe | 每个 OSD 10~30G |
| **Block** | 实际的对象数据 | HDD/SSD | 剩下所有空间 |

### 1.3 为什么要分离 DB 和 WAL 到 SSD？

**场景：HDD 做数据盘，SSD 做 DB/WAL 加速**

| 配置 | 随机写 IOPS | 提升 |
|------|------------|------|
| 全在 HDD | ~100 | 基准 |
| DB 放 SSD，WAL 放 HDD | ~500 | 5 倍 |
| DB + WAL 都放 SSD | ~1500 | 15 倍 |
| DB + WAL 都放 NVMe | ~5000 | 50 倍 |

> 💡 这是性价比最高的优化方式！
> 用 1 块 1T 的 NVMe，可以给 30 个 HDD OSD 做 DB/WAL 加速，性能提升一个数量级。

### 1.4 BlueStore 写入流程

```
客户端写入请求
    ↓
1. 写入 WAL（刷盘，保证不丢）
    ↓
2. 返回写入成功给客户端  ← 这里就返回了！
    ↓ 后台异步
3. 把元数据写入 RocksDB
    ↓
4. 把数据写入 Block 区域
    ↓
5. 定期做 Compaction，整理空间
```

**关键理解：**
- WAL 刷盘成功就返回，延迟很低
- 实际的数据写入是后台异步的
- RocksDB 的 LSM 树结构很适合这种写入模式

### 1.5 BlueStore 内存开销

BlueStore 很吃内存，这是很多人踩过的坑：

| 配置 | 每个 OSD 内存 |
|------|-------------|
| 最小可用 | 2G |
| 推荐配置 | 4G |
| 良好性能 | 8G |

> 🚨 内存不足的后果：
> RocksDB 性能急剧下降，BlueStore 会把大量元数据 swap 到磁盘，整个 OSD 慢成狗。
> 所以一定要给够内存！

## 二、一致性与可靠性机制

### 2.1 三副本写入一致性

```
客户端 → 主 OSD
    ↓ 同时写
主 OSD → 从 OSD 1
主 OSD → 从 OSD 2
    ↓ 两个从都返回成功
主 OSD 记录到自己的 PG 日志
    ↓
返回成功给客户端
```

**关键点：**
- 必须所有从 OSD 都写成功，主才返回成功
- 只要有一个从写失败，PG 就进入 degraded 状态
- PG 日志里记录了所有操作，用来恢复一致性

### 2.2 Scrub 机制

Scrub 是 Ceph 的"数据体检"机制：

| 类型 | 检查内容 | 频率 | 影响 |
|------|---------|------|------|
| **Scrub（浅 scrub）** | 检查对象存在性、大小 | 每天一次 | 轻量，几乎不影响 |
| **Deep Scrub（深 scrub）** | 读取所有数据，校验 checksum | 每周一次 | 较重，会读所有数据 |

**Scrub 发现不一致怎么办？**
1. 标记 PG 为 inconsistent 状态
2. 不会自动修复，等你确认
3. 你确认后，用 `ceph pg repair <pgid>` 修复
4. Ceph 会用正确的副本覆盖错误的副本

> 💡 Deep Scrub 非常重要！
> 磁盘会有静默损坏（bit rot），数据在那好好的，读出来就错了。
> Deep Scrub 是唯一能检测出这种问题的机制。

### 2.3 静默数据损坏的防护

Ceph 对数据完整性的防护是端到端的：

1. **写入时**：计算每个对象的 checksum
2. **存储时**：checksum 和数据一起存在磁盘上
3. **读取时**：重新计算 checksum，和存的对比
4. **Scrub 时**：所有副本的 checksum 互相对比

如果发现不一致：
- 自动用正确的副本返回给客户端
- 后台修复错误的副本

## 三、性能调优原理

### 3.1 性能瓶颈排查思路

按这个顺序排查，90% 的性能问题都能找到：

```
1. 先看网络
   → 有没有丢包？延迟多少？带宽打满了吗？
   → MTU 对不对？有没有半开连接？

2. 再看磁盘
   → iostat 看 %util 是不是 100%？
   → await 是不是超过 20ms？
   → 是不是 HDD 的 IOPS 到上限了？

3. 再看 CPU
   → 是不是 CPU 100% 了？
   → 是不是单个 OSD 进程占满一个核了？

4. 最后看 Ceph 本身
   → PG 是不是在 peering？
   → 有没有在重平衡？
   → BlueStore 是不是在做 compaction？
```

### 3.2 最有效的几个调优参数

| 参数 | 默认值 | 推荐值 | 效果 |
|------|--------|--------|------|
| `osd_memory_target` | 4G | 8G | 给更多内存，RocksDB 缓存更大 |
| `osd_max_backfills` | 1 | 4 | 恢复速度更快，不要太大 |
| `osd_recovery_max_active` | 3 | 10 | 恢复并发更大 |
| `bluestore_cache_size` | 1G | 2G | BlueStore 数据缓存更大 |

> 💡 不要上来就乱调几十个参数！
> 大部分参数默认值就很好，调对上面这 4 个，就能解决 80% 的性能问题。

### 3.3 全闪集群的特殊调优

全闪和混闪是完全不同的调优思路：

- 关闭节流：SSD 不怕高并发，把 recovery 相关的限制都放开
- 加大队列深度：SSD 的队列深度可以开到 32 以上
- 关闭不必要的校验：SSD 本身有 ECC，有些校验可以关
- 调整 RocksDB 参数：针对 NVMe 做专门优化

## 四、缓存分层（Cache Tier）

### 4.1 为什么不推荐用 Cache Tier 了？

Cache Tier 曾经是"银弹"，用 SSD 给 HDD 做缓存层，听起来很美好。

但实际用下来 90% 的场景都是负优化：

❌ **问题 1：一致性开销大**
- 写入要先写缓存层，再刷回后端层
- 还要保证副本一致，开销很大

❌ **问题 2：冷热数据识别不准**
- Ceph 的 LRU 算法很一般，经常把冷数据放缓存，热数据踢出去
- 结果就是缓存命中率很低，还不如直接读

❌ **问题 3：故障域变大**
- 缓存层挂了，整个池都挂了
- 缓存层满了，整个池都写不进去

### 4.2 更好的替代方案

不要用 Cache Tier，用这两个方案：

1. **分离 DB/WAL 到 SSD**
   - 前面说过了，性价比最高，没有一致性问题

2. **直接建全闪池**
   - 热数据放全闪池，冷数据放 HDD 池
   - 自己在业务层做冷热分离，比 Ceph 自动的靠谱多了

> 🚨 官方现在也不推荐用 Cache Tier 了，新部署不要用！

## 五、故障排查高级技巧

### 5.1 PG 状态异常排查

```bash
# 看哪个 PG 有问题
ceph health detail

# 看这个 PG 的详细状态
ceph pg dump pglog | grep <pgid>

# 看这个 PG 在哪几个 OSD 上
ceph pg map <pgid>

# 查看这几个 OSD 的日志
journalctl -u ceph-osd@<osdid> -f

# 如果是 inconsistent，就修复
ceph pg repair <pgid>
```

### 5.2 OSD 挂了怎么办？

按这个顺序来：

1. **先看为什么挂**
   ```bash
   journalctl -u ceph-osd@0 --since "1 hour ago"
   ```
   - 是磁盘坏了？看 smartctl
   - 是 OOM 了？看 dmesg
   - 是网络断了？ping 一下

2. **尝试重启**
   ```bash
   systemctl restart ceph-osd@0
   ```
   - 很多时候重启就好了

3. **真的坏了？换盘**
   - 先 `ceph osd out 0`，让数据迁走
   - 换完新盘，重新创建 OSD
   - `ceph osd in 0`，加回来

### 5.3 慢请求排查

```bash
# 看有没有慢请求
ceph osd perf

# 看是哪个 OSD 慢
ceph daemon osd.0 perf dump

# 看这个 OSD 上的慢操作
ceph daemon osd.0 dump_ops_in_flight
```

慢请求 90% 的原因：
- HDD 的 IOPS 到上限了
- DB/WAL 没放 SSD，RocksDB 慢了
- 网络延迟高或者丢包
- BlueStore 在做大的 compaction
