# Ceph 生产集群部署架构指南

> 这是最容易踩坑的部分：90% 的生产问题，在部署规划阶段就已经埋下了。
> 大部分教程只会教你怎么装，不会告诉你"为什么这么设计"。

---

## 一、集群规模分级

| 规模 | 节点数 | OSD 数量 | 适用场景 |
|-----|-------|---------|---------|
| **微型** | 3 节点 | 6~12 OSD | 开发测试、POC 验证 |
| **小型** | 5~8 节点 | 20~60 OSD | 100TB 级，中小团队生产 |
| **中型** | 10~20 节点 | 60~200 OSD | PB 级，企业级生产 |
| **大型** | 30+ 节点 | 200+ OSD | 10PB 级，云服务商规模 |

> ⚠️ **绝对不要在生产用 3 节点集群！**
> 
> 3 节点是能跑 Ceph 的最小配置，但不是生产配置：
> - 坏一个节点，PG 降级，性能暴跌
> - 升级需要重启节点，期间只剩 2 个，再坏一个就挂了
> - MON 同时也是 OSD，互相影响
> 
> **生产最小配置：5 节点起步！**

## 二、节点角色分离架构

### 2.1 理想的角色分离（中型以上集群）

`
┌─────────────────────────────────────────────────────────┐
│                    管控平面节点 (3~5台)                   │
│   MON + MGR + MDS + RGW                                  │
│   低CPU、低内存、SSD 系统盘                               │
│   不跑 OSD！不跑业务！                                    │
└───────────────────┬─────────────────────────────────────┘
                    │
┌───────────────────┴─────────────────────────────────────┐
│                    存储节点 (N台)                         │
│   OSD + BlueStore                                        │
│   高CPU、大内存、HDD/SSD 数据盘                           │
│   只跑存储，不干别的                                      │
└───────────────────┬─────────────────────────────────────┘
                    │
┌───────────────────┴─────────────────────────────────────┐
│                    客户端节点                             │
│   K8s 节点、虚拟机、应用服务器                             │
│   只做 RBD/RGW 客户端                                     │
└─────────────────────────────────────────────────────────┘
`

### 2.2 退而求其次（小型集群）

如果预算有限，至少要做到：
- **MON 节点不跑 OSD**（管控和存储分离）
- **RGW 节点和 OSD 节点分开**（对象存储很吃带宽）

> ❌ 最糟糕的架构："All in One"
> 
> 3 台机器，每台同时跑 MON + MGR + OSD + RGW + 虚拟机业务
> 这是新手最常见的坑，出了问题互相影响，根本查不出是哪层的锅。

## 三、硬件选型指南

### 3.1 CPU 配置

| 角色 | CPU 要求 | 举例 |
|-----|---------|------|
| **MON** | 低 | 8C 够了，MON 基本不耗 CPU |
| **MGR** | 低 | 8C 够了，监控面板费一点 |
| **OSD (HDD)** | 中 | 每个 OSD 配 1 个物理核心 → 12 盘 = 12C |
| **OSD (SSD/NVMe)** | 高 | 每个 OSD 配 2 个物理核心 → 12 盘 = 24C |
| **RGW** | 高 | 对象存储非常吃 CPU，至少 16C 起步 |
| **EC 纠删码** | 很高 | 纠删码计算量是副本的 3~5 倍 |

> 💡 CPU 不够 = 延迟高。Ceph 是 CPU 密集型，不是 IO 密集型，很多人搞反了。

### 3.2 内存配置

| 角色 | 内存要求 | 举例 |
|-----|---------|------|
| **MON** | 低 | 16G 够了 |
| **MGR** | 中 | 32G，Prometheus 吃内存 |
| **OSD (HDD)** | 每个 OSD 4G | 12 盘 HDD = 48G 内存 |
| **OSD (SSD)** | 每个 OSD 8G | 12 盘 SSD = 96G 内存 |
| **OSD (NVMe)** | 每个 OSD 16G | 全闪节点建议 256G 起步 |

> 🚨 内存不足 = OSD 自动 crash。
> BlueStore 很吃内存缓存，内存给够是性价比最高的优化。

### 3.3 磁盘选型

| 用途 | 介质 | 容量建议 | 备注 |
|-----|------|---------|------|
| **系统盘** | SATA SSD | 256G~512G | 别用 HDD 做系统盘，OSD 启动慢会被踢 |
| **HDD 数据盘** | SATA HDD | 8T~18T | 主容量，性价比高，建议同型号同容量 |
| **SSD 数据盘** | SATA/SAS SSD | 2T~8T | 高性能数据池 |
| **NVMe 数据盘** | NVMe SSD | 1.6T~6.4T | 全闪池，RGW 元数据池 |
| **DB/WAL 加速** | NVMe | 每个 OSD 分 10~30G | 混闪架构必备，性能提升 5~10 倍 |

> ❌ 不要在 Ceph 里用 RAID！
> Ceph 自己做多副本，RAID 纯属画蛇添足，还拖慢性能。
> HBA 卡直通单盘是最佳实践。

## 四、网络架构（最最最重要！）

Ceph 的性能瓶颈 80% 出在网络上。

### 4.1 双网卡分离（生产必须！）

`
              ┌───────────────────────────────────────┐
公网 10G/25G  │  业务流量：客户端读写 RBD/RGW          │
              └───────────────────┬───────────────────┘
                                  │
              ┌───────────────────┴───────────────────┐
集群网 10G/25G│  内部流量：OSD 复制、恢复、重平衡        │
              └───────────────────────────────────────┘
`

**为什么必须分离？**
- 集群重平衡时会打满带宽，如果和公网共用，业务直接卡死
- 复制流量是东-west，业务流量是南-north，特点不一样
- 故障隔离：公网出问题不影响数据一致性

### 4.2 带宽选型

| 集群规模 | 公网 | 集群网 |
|---------|------|--------|
| 微型测试 | 1G | 1G |
| 小型生产 | 10G | 10G |
| 中型生产 | 10G/25G | 25G |
| 大型全闪 | 25G/100G | 100G |

> 💡 集群网带宽建议 ≥ 公网带宽。
> 三副本写一份数据，内网要跑两份副本流量。

### 4.3 网络优化要点

1. **MTU 9000 巨帧**：整条链路交换机和网卡都要开，TCP 包数量减 6 倍
2. **Bond Mode 4 (LACP)**：网卡绑定，balance-xor 或 layer3+4 哈希
3. **单独的交换机**：不要和别的业务共用交换机，Ceph 流量很"暴"
4. **不要跨三层网络部署 OSD**：MON 可以跨机房，OSD 尽量在同一个二层

## 五、机架与故障域规划

### 5.1 标准的三级故障域

`
root default
├── rack1
│   ├── host01
│   ├── host02
│   └── host03
├── rack2
│   ├── host04
│   ├── host05
│   └── host06
└── rack3
    ├── host07
    ├── host08
    └── host09
`

**三副本 + rack 级故障域 = 整个机架断电，数据不丢，业务不中断。**

### 5.2 PG 分布约束

- 同一个 PG 的 3 个副本，绝对不能在同一个主机
- 最好也不要在同一个机架（如果有 3 个以上机架）
- 机房级容灾：3 个机房，每个机房放一个副本

## 六、不同场景的参考配置

### 6.1 小型生产集群（5 节点）

| 节点 | CPU | 内存 | 磁盘 | 角色 |
|-----|-----|------|------|------|
| node1~3 | 8C | 32G | 系统盘 500G SSD | MON + MGR |
| node4~5 | 16C | 64G | 12 块 8T HDD + 1 块 NVMe 做 DB/WAL | OSD 存储节点 |

> 总共 24 个 OSD，三副本可用容量 ~ 64T，适合 100 人以内团队。

### 6.2 中型生产集群（12 节点）

| 节点数 | CPU | 内存 | 磁盘 | 角色 |
|-------|-----|------|------|------|
| 5 台 | 8C | 32G | 500G SSD | MON + MGR + RGW |
| 7 台 | 24C | 128G | 12×16T HDD + 1×1.6T NVMe | OSD 存储节点 |

> 总共 84 个 OSD，三副本可用 ~ 450T，适合企业级 K8s 集群后端。

### 6.3 全闪高性能集群（8 节点）

| 节点数 | CPU | 内存 | 磁盘 | 角色 |
|-------|-----|------|------|------|
| 3 台 | 16C | 64G | 1T NVMe | MON + MGR |
| 5 台 | 32C | 256G | 8×3.2T NVMe | 全闪 OSD |

> 总共 40 个 NVMe OSD，三副本可用 ~ 130T，适合数据库、高性能虚拟化场景。

## 七、架构避坑清单

✅ **一定要做的：**
- [ ] 节点角色分离，MON 不跑 OSD
- [ ] 双网卡分离公网和集群网
- [ ] 所有 OSD 同型号同容量（异构容量会导致利用率不均）
- [ ] 内存给够，每个 OSD 至少 4G
- [ ] 至少 3 个机架，做机架级故障域
- [ ] 时间同步！必须精确到毫秒级

❌ **绝对不要做的：**
- [ ] 3 节点跑生产
- [ ] MON 和 OSD 跑在同一台机器
- [ ] 公网和集群网共用网卡
- [ ] 给 OSD 做 RAID5/RAID6
- [ ] 不同速度的盘混在同一个 CRUSH 根
- [ ] 单个 OSD 容量超过 18T（恢复太慢）

---

## 八、一句话总结

> **Ceph 生产架构的精髓，不是"怎么跑起来"，而是"怎么在各种故障下还能跑"。**
>
> 硬件冗余、网络隔离、故障域规划，这些才是你半夜能睡好觉的保障。
