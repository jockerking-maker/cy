# Ceph 常见故障处理实战手册

> 出问题别慌，按这个手册一步步来。90% 的故障都是常见问题。

---

## 一、排障总原则

✅ **先保业务，再查根因**
- 先让集群能读写，再慢慢研究为什么坏的

✅ **不要急着重启 OSD**
- 很多时候重启没用，反而会触发更多 peering 让业务更卡

✅ **不要乱删 PG 日志**
- 那是排查问题的唯一线索，删了就真的找不回来了

✅ **先隔离再修复**
- 坏盘先拔了，让集群先恢复，再慢慢研究盘为什么坏

---

## 二、故障排查标准流程

`
ceph -s 看状态
    ↓
哪里红了看哪里
    ├─ OSD down → 去对应节点看日志
    ├─ PG 异常 → 看是哪些 PG，在哪些 OSD 上
    └─ 满了 → 看哪个 OSD 满了
    ↓
看日志 journalctl -u ceph-osd@x
    ↓
找 ERROR 关键词
    ↓
定位根因，修复
`

## 三、TOP 10 常见故障处理

---

### 🚨 故障 1：单个 OSD 自动 crash 反复起不来

**现象**：ceph -s 显示 1 osd down，重启后几分钟又挂了

**排查步骤**：
1. 看 OSD 日志：journalctl -u ceph-osd@12 -f --since "1 hour ago"
2. 找关键词：
   - ad_alloc → **内存不足**，给 OSD 加内存或者调小 osd_memory_target
   - ENOSPC → **磁盘满了**，赶紧删数据扩容
   - Input/output error → **磁盘坏了**，直接换盘
3. 看 dmesg：是不是内核层面的 IO 错误

**快速恢复**：
- 内存不够：ceph config set osd.12 osd_memory_target 4294967296（设为 4G）
- 磁盘坏了：直接 ceph osd out 12，换完新盘再加回来

---

### 🚨 故障 2：集群卡在 peering，PG 一直不 active

**现象**：ceph -s 显示 XX pgs peering，持续几分钟都没变

**原因**：
- PG 所在的多个 OSD 同时挂了，找不到完整的副本
- 网络不通，OSD 之间连不上
- OSD 版本不一致

**排查步骤**：
1. 找卡在 peering 的 PG：ceph pg dump | grep peering
2. 看这个 PG 分布在哪些 OSD 上：ceph pg map <pgid>
3. 检查这些 OSD 是不是都 up，网络能不能通
4. 看主 OSD 的日志：ceph daemon osd.x log last

**快速恢复**：
- 最简单粗暴：ceph osd repair <pgid>
- 还是不行：ceph pg force_create_pg <pgid>（不到万不得已别用）

---

### 🚨 故障 3：PG inconsistent，数据不一致

**现象**：HEALTH_WARN，提示 X pgs inconsistent

**原因**：
- 静默磁盘坏道
- 掉电导致的半写
- scrub 时检测出多个副本内容不一样

**处理步骤**：
1. 找出不一致的 PG：ados list-inconsistent-pg <poolname>
2. 看具体哪个对象坏了：ceph pg <pgid> query | grep inconsistent
3. 执行修复：ceph pg repair <pgid>
4. 等几分钟再看状态

**注意**：
- repair 会用主 OSD 的内容覆盖从 OSD，99% 的场景没问题
- 极端情况主副本也是坏的，那就只能从备份恢复了

---

### 🚨 故障 4：OSD full，集群只读写不进去

**现象**：客户端报 Read-only file system，ceph -s 显示 FULL

**原因**：
- 某个 OSD 使用率到了 osd_full_ratio（默认 95%）
- 哪怕其他 OSD 还有空间，只要一个 OSD 满了，整个集群就写不进去

**紧急处理**：
1. 先临时调高阈值让业务能写：
   `ash
   ceph osd set-full-ratio 0.97
   ceph osd set-backfillfull-ratio 0.96
   `
2. 赶紧删东西：删没用的快照、删旧的 RBD 镜像
3. 长期解决方案：加 OSD 扩容

> ⚠️ 这个故障非常危险！满了之后恢复也很慢，一定要提前监控容量。

---

### 🚨 故障 5：重平衡/恢复太慢，业务卡爆

**现象**：加了新 OSD 或者坏了一个盘，恢复跑了好几天，业务卡成狗

**原因**：默认恢复速度参数太保守

**调优方案**：
`ash
# 临时加速恢复
ceph config set global osd_recovery_op_priority 30
ceph config set global osd_max_backfills 10
ceph config set global osd_recovery_max_active 10

# 恢复完了记得改回去！
ceph config rm global osd_recovery_op_priority
ceph config rm global osd_max_backfills
ceph config rm global osd_recovery_max_active
`

**注意**：别调太大，不然恢复带宽把业务带宽吃光了，业务更卡。

---

### 🚨 故障 6：MON 仲裁失败，集群挂了

**现象**：所有命令都卡住，ceph -s 半天没反应

**原因**：
- 3 个 MON 挂了 2 个，达不到法定人数
- MON 之间网络不通
- MON 磁盘满了

**处理步骤**：
1. 看哪些 MON 还活着：ceph mon stat
2. 去挂了的 MON 节点，看 mon 日志
3. 至少要起来一半以上的 MON 才能恢复
4. 真的起不来就删了坏的 MON，加新的 MON 进去

> 💡 MON 建议至少部署 3 个，分布在不同的机架/机房。

---

### 🚨 故障 7：Deep Scrub 跑的时候业务卡

**现象**：每天固定时间业务延迟突然飙升，过几个小时自己好

**原因**：Deep Scrub 要读整个 PG 的所有数据校验，HDD 扛不住

**解决方案**：
1. 错开业务高峰跑，只在凌晨跑：
   `ash
   ceph config set osd osd_scrub_begin_hour 2
   ceph config set osd osd_scrub_end_hour 6
   `
2. 降低 scrub 优先级：
   `ash
   ceph config set osd osd_scrub_sleep 0.1
   `
3. 延长 scrub 间隔，从每周改到每两周：
   `ash
   ceph config set osd osd_deep_scrub_interval 1209600
   `

---

### 🚨 故障 8：RBD 挂载不上，超时

**现象**：bd map 命令卡住或者超时

**排查步骤**：
1. 先看集群状态是不是正常
2. 看客户端到 MON、OSD 的网络通不通，端口 6789、6800-7300
3. 看 RBD 镜像有没有被锁：bd lock list pool/image
4. 有锁的话释放：bd lock remove pool/image locker cookie

**常见原因**：
- 客户端节点异常重启，RBD 锁没释放
- 网络不通或者 MTU 不匹配
- 内核版本太老有 bug

---

### 🚨 故障 9：集群写入性能突然暴跌

**现象**：以前写 1G/s，现在只有 100M/s

**排查顺序**（按概率从高到低）：
1. 是不是在恢复/重平衡？ceph -s 看有没有 backfilling
2. 是不是在跑 deep scrub？ceph pg dump | grep scrub
3. 看哪个 OSD 延迟高：ceph osd perf
4. 看那个 OSD 的磁盘是不是坏道了：smartctl -a /dev/sdx
5. 看网络有没有丢包：ethtool -S eth0 | grep error
6. 看 CPU 是不是跑满了

90% 的性能问题都出在「某个 OSD 磁盘坏了变慢，拖慢整个集群」。

---

### 🚨 故障 10：删了东西空间没释放

**现象**：删了好几个 TB 的 RBD 镜像，ceph df 显示可用空间没变

**原因**：
- RBD 镜像只是标记删除，实际数据要等垃圾回收
- 默认回收间隔是 24 小时

**手动加速**：
`ash
# 立即触发垃圾回收
rbd trash purge poolname --all

# 看回收站里有什么
rbd trash ls poolname
`

---

## 四、数据丢失了怎么办？

先别慌，按这个顺序来：

1. **停写！** 不要往集群写新东西，避免覆盖
2. 看 PG 状态，是不是真的所有副本都没了
3. 看 OSD 是不是只是 down 了，数据还在磁盘上
4. 尝试把 OSD 手动 mount 出来，导数据
5. 真的不行就从备份恢复

> 💡 Ceph 三副本配置下，真·数据丢失的概率极低。
> 大部分"数据丢了"的情况，都是 PG 暂时找不到副本，等 OSD 起来就好了。

## 五、故障处理避坑清单

❌ **绝对不要做的事：**
- [ ] 不要 ceph osd rm 一个还有数据的 OSD，除非你确定数据都迁走了
- [ ] 不要同时重启超过 1 个 OSD，不然 PG 找不到副本
- [ ] 不要随便 ceph osd crush reweight，会触发全集群重平衡
- [ ] 不要删 PG 日志，那是排查问题的唯一线索
- [ ] 生产环境不要随便升级版本，小版本也可能出大问题

✅ **一定要做的事：**
- [ ] 操作前先拍个 ceph report 留档
- [ ] 大操作先在测试集群验证
- [ ] 改参数前先记一下原来的值，方便回滚
- [ ] 有备份！有备份！有备份！

## 六、一句话总结

> **Ceph 很健壮，但也很脆。大部分故障不是 Ceph 本身的问题，而是硬件、网络、或者人瞎操作搞出来的。**
> 
> 按手册一步步来，别慌，90% 的问题都能修好。
