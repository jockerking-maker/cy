# 第二阶段：入门基础详解

> 目标：能独立搭建集群，看懂所有状态，完成基本操作。

---

## 一、Ceph 核心设计哲学

### 1.1 Ceph 要解决的核心问题

传统存储的痛点：
- ❌ 容量到上限了？买新柜子，数据迁移要死人
- ❌ 性能到瓶颈了？纵向升级，贵得要死
- ❌ 控制器坏了？整个存储集群都挂了
- ❌ 厂商锁定？想换？先把所有数据拷出来

Ceph 的解决方案：
- ✅ 横向扩展：加服务器就能扩容和加性能
- ✅ 无单点故障：任何组件挂了都不影响全集群
- ✅ 统一存储：块、对象、文件一套搞定
- ✅ 开放标准：没有厂商锁定

### 1.2 三大设计原则

1. **所有组件必须可扩展**
   - 没有中心节点，没有元数据瓶颈
   - 任何组件都可以水平扩展

2. **必须处理故障，而不是避免故障**
   - 硬件故障是常态，不是意外
   - 系统必须能自动检测、自动修复

3. **所有算法必须可并行**
   - 避免全局锁
   - 每个操作都应该能分布式执行

## 二、五大核心组件

### 2.1 MON（Monitor）

**角色**：集群的"大脑"和"身份证管理局"

**职责**：
- 维护集群的所有 Map（OSD Map、MON Map、PG Map、CRUSH Map）
- 集群认证管理（cephx 密钥）
- 集群配置管理

**关键特性**：
- 必须奇数台：3、5、7... 最多 7 台就够了
- 用 Paxos 算法选主，只要过半活着集群就正常
- 3 台 MON，允许挂 1 台；5 台 MON，允许挂 2 台
- 不处理任何业务 IO，只做元数据管理

**配置建议**：
- CPU：2C 够了，不怎么耗
- 内存：8G 起步，16G 保险
- 磁盘：SSD，不需要很大，100G 就够
- 网络：一定要低延迟，MON 对延迟很敏感

> 💡 不要把 MON 部署在 OSD 同一台服务器上！
> MON 挂了不丢数据，但 OSD 挂了可能丢数据，两者要物理分离。

### 2.2 OSD（Object Storage Device）

**角色**：集群的"搬运工"，真正干活的

**职责**：
- 存储实际数据对象
- 处理客户端的读写请求
- 数据复制、恢复、重平衡
- 心跳检测，告诉 MON 自己还活着

**关键特性**：
- 一块盘 = 一个 OSD（推荐）
- 每个 OSD 都有自己的独立进程
- 三副本场景下，一个对象写入 3 个不同的 OSD

**配置建议**：
- CPU：每个 OSD 至少 1 个物理核心
- 内存：每个 OSD 至少 4G，推荐 8G（BlueStore 很吃内存）
- 网络：双网卡，公网 + 集群网分离

### 2.3 MGR（Manager）

**角色**：集群的"仪表盘"和"管理接口"

**职责**：
- REST API 接口
- Prometheus 监控指标导出
- Dashboard Web 界面
- 各种插件（自动均衡、设备健康等）

**关键特性**：
- 主备模式，同一时间只有一个 Active MGR
- MGR 挂了不影响数据面，只是监控和管理用不了
- 至少部署 2 个 MGR 做高可用

### 2.4 MDS（Metadata Server）

**角色**：只有用 CephFS 文件存储才需要

**职责**：管理 CephFS 的目录树、文件元数据

**注意**：不用 CephFS 的话，完全不需要部署 MDS。

### 2.5 RGW（RADOS Gateway）

**角色**：只有用对象存储才需要

**职责**：提供 S3 兼容的 HTTP API 接口

**注意**：不用对象存储的话，完全不需要部署 RGW。

## 三、第一次部署集群

### 3.1 环境准备

**推荐的最小测试集群：**
| 节点 | 配置 | 磁盘 | 网络 |
|------|------|------|------|
| node1 | 2C 8G | 系统盘 + 2 块空盘 | 公网 + 集群网 |
| node2 | 2C 8G | 系统盘 + 2 块空盘 | 公网 + 集群网 |
| node3 | 2C 8G | 系统盘 + 2 块空盘 | 公网 + 集群网 |

> 总共 6 个 OSD，三副本正好够用。

**前置配置（所有节点）：**
```bash
# 1. 配置主机名和 hosts，互相能解析
# 2. 关闭防火墙和 SELinux（测试环境）
# 3. 配置 SSH 免密登录
# 4. 时间同步！非常重要！ceph 对时间差容忍度很低
# 5. 配置 yum/apt 源
```

### 3.2 用 cephadm 部署（推荐）

不要用 ansible、不要用 rook 入门，就用官方的 cephadm：

```bash
# 1. 在 node1 上引导集群
curl --silent --remote-name --location https://github.com/ceph/ceph/raw/reef/src/cephadm/cephadm
chmod +x cephadm
./cephadm bootstrap --mon-ip 192.168.1.101

# 2. 把另外两个节点加进去
ceph orch host add node2
ceph orch host add node3

# 3. 部署 OSD
ceph orch daemon add osd node1:/dev/sdb
ceph orch daemon add osd node1:/dev/sdc
# ... 6 块盘都加完

# 4. 查看状态
ceph -s
```

看到 `HEALTH_OK`，恭喜你，集群搭成了！🎉

## 四、必背的基础命令

### 4.1 集群状态

```bash
ceph -s                      # 最常用，看整体状态
ceph status                  # 和上面一样
ceph health detail           # 健康状态的详细信息
ceph df                      # 集群空间使用情况
ceph osd df                  # 每个 OSD 的使用率
ceph osd tree                # OSD 的 CRUSH 分布树
ceph pg stat                 # PG 状态统计
```

### 4.2 OSD 操作

```bash
ceph osd ls                  # 列出所有 OSD
ceph osd stat                # OSD 统计
ceph osd down 0              # 标记 OSD 0 为 down
ceph osd out 0               # 把 OSD 0 踢出集群（数据会迁移走）
ceph osd in 0                # 把 OSD 0 加回来
ceph osd rm 0                # 删除 OSD 0（数据都迁走了才能删）
```

### 4.3 RBD 块存储操作

```bash
# 创建存储池
ceph osd pool create rbd_pool 128 128

# 启用 RBD 应用
ceph osd pool application enable rbd_pool rbd

# 创建 10G 的 RBD 镜像
rbd create --size 10G --pool rbd_pool my_image

# 列出所有 RBD 镜像
rbd ls -l rbd_pool

# 映射到本地
rbd map rbd_pool/my_image

# 格式化挂载
mkfs.xfs /dev/rbd0
mount /dev/rbd0 /mnt
```

## 五、看懂集群状态

### 5.1 ceph -s 输出解读

```
  cluster:
    id:     f8a83e4a-2d1b-4c3d-8e7f-9a0b1c2d3e4f
    health: HEALTH_OK        ← 这就是我们要的！

  services:
    mon: 3 daemons, quorum node1,node2,node3 (age 2h)
    mgr: node1(active, since 2h), standbys: node2, node3
    osd: 6 osds: 6 up (since 2h), 6 in (since 2h)

  data:
    pools:   1 pools, 128 pgs
    objects: 0 objects, 0 B
    usage:   6.0 TiB used, 54 TiB / 60 TiB avail
    pgs:     128 active+clean
```

### 5.2 健康状态分级

| 状态 | 含义 | 怎么办 |
|------|------|--------|
| **HEALTH_OK** | 一切正常 | 喝茶 |
| **HEALTH_WARN** | 有问题，但还不影响数据 | 有空了看看 |
| **HEALTH_ERR** | 有严重问题，可能影响数据 | 立刻处理 |
