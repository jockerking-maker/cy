## 一、Linux 系统基础

### 1.1 磁盘管理

#### 必须掌握的命令：
```bash
lsblk                  # 查看块设备列表，找空盘
fdisk -l               # 查看磁盘分区
parted /dev/sdb mklabel gpt  # 创建 GPT 分区表（Ceph 推荐）
dd if=/dev/zero of=/dev/sdb bs=1M count=100  # 擦除磁盘头部
hdparm -tT /dev/sda    # 测试磁盘读写速度
smartctl -a /dev/sda   # 查看磁盘健康状态（必须会！）
```

#### 关键知识点：
- **为什么 Ceph 不用 RAID？**
  - Ceph 本身就是多副本，RAID 是重复建设
  - RAID 卡本身就是单点故障
  - RAID 的写惩罚反而拖慢性能
  - Ceph 的恢复粒度比 RAID 细得多

- **磁盘分区注意事项：**
  - 不要给 OSD 数据盘分区！Ceph 会接管整块盘
  - BlueStore 的 DB 和 WAL 可以单独分区到 SSD
  - 一定要用 GPT，不要用 MBR

### 1.2 文件系统

| 文件系统 | 适用场景 | 推荐度 |
|---------|---------|--------|
| **XFS** | 操作系统盘，RGW 日志盘 | ⭐⭐⭐⭐⭐ 推荐 |
| **ext4** | 可以用，但 XFS 更好 | ⭐⭐⭐ |
| **btrfs / zfs** | 不要用，Ceph 官方不推荐 | ❌ 避免 |

#### 关键的 mount 选项：
```
/dev/sda1 / xfs defaults,noatime,nodiratime 0 0
```
- `noatime`：关闭访问时间记录，提升磁盘性能
- `nodiratime`：关闭目录访问时间记录

### 1.3 系统服务与日志

```bash
systemctl status ceph.target       # 查看所有 Ceph 服务状态
systemctl restart ceph-osd@0       # 重启 OSD 0
journalctl -u ceph-osd@0 -f        # 实时查看 OSD 0 的日志（排障必备！）
journalctl --vacuum-size=1G        # 清理日志，避免占满磁盘
```

## 二、存储基础概念

### 2.1 三种存储类型的本质区别

| 类型 | 访问方式 | 代表产品 | 适用场景 | Ceph 对应 |
|------|---------|---------|---------|----------|
| **块存储** | 像本地硬盘一样，格式化后挂载 | SAN、云硬盘 | 虚拟机、数据库、K8s PV | RBD |
| **对象存储** | HTTP API 访问，GET/PUT/DELETE | S3、OSS | 图片、视频、备份、静态文件 | RGW |
| **文件存储** | POSIX 兼容，像网络共享文件夹 | NAS、NFS | 日志收集、共享目录、媒体素材 | CephFS |

### 2.2 HDD vs SSD vs NVMe

| 介质 | 延迟 | IOPS | 带宽 | 价格 | Ceph 用途 |
|------|------|------|------|------|----------|
| HDD | ~10ms | 100~200 | ~200MB/s | 便宜 | 数据盘 |
| SATA SSD | ~0.5ms | 5k~10k | ~500MB/s | 中 | DB/WAL 加速盘 |
| NVMe SSD | ~0.1ms | 50k~100k | ~3GB/s | 贵 | 高性能数据盘、全闪集群 |

> 💡 一个重要认知：
> HDD 的瓶颈是 **IOPS**（每秒能处理多少请求），不是带宽。
> 几百 MB/s 的带宽看起来很高，但随机 IO 只有 100 多，这才是慢的根源。

## 三、网络基础

### 3.1 双网络分离设计（生产必须！）

```
┌─────────────┐
│   公网      │  ← 业务服务器访问 Ceph 用，10G 起步
│ (Public)    │
└─────────────┘

┌─────────────┐
│   集群网    │  ← OSD 之间数据同步、重平衡用，必须和公网分离
│ (Cluster)   │
└─────────────┘
```

**为什么必须分离？**

❌ 不分离的后果：
- 集群重平衡的时候，把带宽打满
- 业务 IO 被数据同步阻塞
- 整个集群看起来"卡死了"

✅ 正确做法：
- 至少两张物理网卡（或者用 SR-IOV 虚拟）
- 公网和集群网走不同的交换机
- 集群网至少和公网同带宽，推荐 1.5~2 倍

### 3.2 网卡绑定 Mode4（LACP）

生产环境必须配置网卡绑定：
```bash
# bond0 配置 Mode 4 (802.3ad LACP)
# xmit_hash_policy = layer3+4 （重要！）
```

为什么要 `layer3+4`？
- 默认的 layer2 哈希，同一个 Ceph 节点之间的流量永远走同一条物理链路
- layer3+4 按源目 IP+端口哈希，能真正把流量散开到多条物理链路上

### 3.3 MTU 巨帧

```
标准 MTU = 1500 字节
巨帧 MTU = 9000 字节
```

**开巨帧的好处：**
- 传输同样大小的数据，TCP 包数量减少 6 倍
- CPU 中断减少，CPU 使用率下降
- 实际吞吐量提升 20%~30%

**注意事项：**
- 整条链路都要开：服务器网卡 → 交换机端口 → 对端服务器
- 一端开一端不开 = 网络不通，而且很难排查
- Ceph 集群网强烈建议开，公网可选

### 3.4 延迟 vs 带宽

很多人搞反了：

| 指标 | 影响什么 | 怎么优化 |
|------|---------|---------|
| **带宽** | 大文件顺序读写的速度 | 加网卡，换更高速率的 |
| **延迟** | 小文件随机 IO 的性能 | 换更好的交换机，减少网络跳数 |

> 💡 Ceph 性能瓶颈往往是延迟，不是带宽。
> OSD 之间 RTT 超过 2ms，性能就会明显下降。

## 四、容器与 K8s 基础

### 4.1 为什么现在 Ceph 都和 K8s 绑定？

- 90% 的新 Ceph 集群都是给 K8s 做存储后端的
- Rook 是当前最主流的 Ceph on K8s 部署方式
- 不懂 K8s，你连现代 Ceph 怎么部署的都看不懂

### 4.2 必须懂的 K8s 存储概念

| 概念 | 说人话 |
|------|--------|
| **PV** | Persistent Volume，Ceph 里划分出来的一块存储空间 |
| **PVC** | Persistent Volume Claim，用户说"我要一块 10G 的盘" |
| **StorageClass** | 存储类，定义"这个 PVC 用什么规格的存储" |
| **CSI** | Container Storage Interface，K8s 和存储系统对接的标准接口 |

### 4.3 基础工作流

```
用户创建 PVC
    ↓
K8s 看 PVC 指定的 StorageClass
    ↓
调用 CSI 驱动去 Ceph 创建 RBD 镜像
    ↓
RBD 映射到 Pod 所在节点，挂载进 Pod
```
