## 一、CRUSH 算法（SSS 级重点）

### 1.1 CRUSH 是什么？

**CRUSH = Controlled Replication Under Scalable Hashing**

这是 Ceph 最核心的发明，也是为什么 Ceph 能无限扩展的根本原因。

#### 传统分布式存储的问题：
```
客户端问元数据服务器："我的数据存在哪？"
元数据服务器查哈希表："存在节点 3、5、7"
客户端去节点 3、5、7 读写
```
❌ 问题：元数据服务器是单点瓶颈，节点增减要搬大量数据。

#### Ceph 的方式：
```
客户端拿到 CRUSH Map
客户端自己算："对象 X 应该存在 OSD 12、25、37"
客户端直接去这三个 OSD 读写
```
✅ 好处：
- 没有元数据瓶颈，不需要问任何人
- 节点增减只需要搬最少的数据
- 所有客户端算法一致，结果一致

> 💡 划重点：CRUSH 不是一致性哈希！
> 一致性哈希是简单的环，CRUSH 是有层级结构的加权哈希。

### 1.2 CRUSH Map 的结构

```
# 层级结构示例
root default
  rack rack1
    host node1
      osd.0
      osd.1
    host node2
      osd.2
      osd.3
  rack rack2
    host node3
      osd.4
      osd.5
    host node4
      osd.6
      osd.7
```

**层级的意义 = 故障域**：
- host 级：同一台主机的 OSD 不会存同一个对象的副本
- rack 级：同一个机架的 OSD 不会存同一个对象的副本
- room 级：同一个机房... 以此类推

> 这就是为什么 Ceph 一个机架断电，数据不会丢——副本本来就不在同一个机架上。

### 1.3 操作 CRUSH Map

```bash
# 导出当前的 CRUSH Map
ceph osd getcrushmap -o crushmap.bin

# 反编译成人类可读的文本
crushtool -d crushmap.bin -o crushmap.txt

# 编辑 crushmap.txt ...

# 编译回去
crushtool -c crushmap.txt -o crushmap-new.bin

# 导入生效
ceph osd setcrushmap -i crushmap-new.bin
```

### 1.4 CRUSH Rule

CRUSH Rule 定义了"数据怎么放"：

```
rule replicated_rule {
    id 0
    type replicated
    min_size 1
    max_size 10
    step take default
    step chooseleaf firstn 0 type host  # 最关键的一行！
    step emit
}
```

**`step chooseleaf firstn 0 type host` 的含义：**
- 在 `default` 根下面选
- 选 N 个不同的 host（N = 副本数）
- 每个 host 里选一个 OSD
- 这 N 个 OSD 就是存副本的位置

**常用的故障域级别：**
- `type osd`：只要求不同 OSD（不推荐，可能在同一台主机）
- `type host`：不同主机（最小生产配置）
- `type rack`：不同机架（推荐，机柜级容灾）
- `type room`：不同机房（两地三中心用）

### 1.5 权重

每个 OSD 都有权重，权重越大，存的数据越多：
- 1T 的盘，权重设为 1.0
- 4T 的盘，权重设为 4.0
- 这样空间使用率自然均衡

```bash
# 调整 OSD 权重
ceph osd reweight osd.0 0.8  # 临时调整，范围 0~1
ceph osd crush reweight osd.0 3.5  # 永久调整 CRUSH 权重
```

## 二、PG 原理（SSS 级重点）

### 2.1 PG 是什么？

**PG = Placement Group，放置组**

你可以把 PG 理解成"逻辑容器"：
- 一个 PG 里装着很多个对象
- 一个 PG 对应一组 OSD（三副本就是 3 个 OSD）
- 一个 OSD 上会有很多个 PG

```
对象 → 哈希 → PG → CRUSH → OSD 组
```

**为什么需要 PG 这一层？不能直接对象映射 OSD？**

如果直接对象→OSD：
- 几千万个对象，每个对象都要追踪状态，元数据爆炸
- OSD 挂了，每个对象都要重新计算位置，开销巨大

有了 PG 这一层：
- 几千个 PG，追踪 PG 状态就行，开销小很多
- OSD 挂了，以 PG 为单位迁移数据，效率高很多

> 💡 划重点：PG 是 Ceph 数据管理、迁移、恢复的基本单位。

### 2.2 PG 数量计算

**官方公式：**
```
PG 总数 = (OSD 总数 × 100) ÷ 副本数
结果向上取最近的 2 的幂
```

**示例：**
- 6 个 OSD，三副本：(6 × 100) ÷ 3 = 200 → 取 256
- 12 个 OSD，三副本：(12 × 100) ÷ 3 = 400 → 取 512

**PG 太多太少都不行：**
| PG 数量 | 问题 |
|---------|------|
| **太少** | 数据分布不均匀，单个 PG 太大，恢复慢 |
| **太多** | 每个 PG 都要占内存和 CPU，资源消耗过大 |

### 2.3 PG 状态机

看到 `active+clean` 你就放心喝茶，看到其他状态要知道是什么意思：

| 状态 | 含义 | 怎么办 |
|------|------|--------|
| **active+clean** | 正常，所有副本都在，数据一致 | ✅ 没问题 |
| **active+degraded** | 有副本挂了，还在恢复中 | ⚠️ 等着，一般会自己好 |
| **active+remapped** | CRUSH 变了，数据正在重平衡 | ⚠️ 等着 |
| **peering** | PG 正在和副本核对状态 | ⚠️ 正常，慢的话要查原因 |
| **active+inconsistent** | 副本之间数据不一致了 | ❗ 需要手动修复 |
| **inactive** | PG 不可用，读写都卡住了 | 🚨 立刻排查！ |

### 2.4 Peering 过程

Peering 是 PG 恢复一致性的核心过程：

1. 主 OSD 找到所有从 OSD
2. 大家对一下各自的 PG 日志
3. 找出缺失的对象，决定谁补谁
4. 补完之后，PG 进入 active 状态

> 💡 Peering 快不快，主要看 PG 日志有多大。
> PG 太大的话，Peering 会非常慢，业务会卡。

## 三、副本 vs 纠删码

### 3.1 三副本机制

**写入流程：**
```
客户端写请求 → 主 OSD
    ↓ 同时写两个从 OSD
主 OSD 收到两个从 OSD 的确认
    ↓
返回成功给客户端
```

**优点：**
- 性能好，写入只需要两跳
- 恢复快，坏一个 OSD 只需要从另外两个拷
- 读性能好，可以从任意副本读

**缺点：**
- 空间利用率只有 33%，太浪费了

### 3.2 纠删码（EC）

纠删码是用"校验块"换空间的技术：

| EC 配置 | 数据块 | 校验块 | 允许坏几块 | 空间利用率 |
|---------|--------|--------|-----------|-----------|
| **4+2** | 4 | 2 | 2 | 4/6 = 67% |
| **8+3** | 8 | 3 | 3 | 8/11 = 73% |
| **12+4** | 12 | 4 | 4 | 12/16 = 75% |

**优点：**
- 空间利用率高很多，冷数据归档非常划算

**缺点：**
- 写入性能差，要算校验码
- 恢复慢，坏一块要读所有数据块来重算
- 不适合 RBD 这种低延迟场景

### 3.3 选型建议

| 场景 | 推荐 |
|------|------|
| 生产环境 RBD 块存储 | ✅ 三副本 |
| 数据库、虚拟机磁盘 | ✅ 三副本 |
| 日志、备份、归档数据 | ✅ 纠删码 4+2 |
| 冷数据、很少访问的 | ✅ 纠删码 8+3 |

## 四、存储三兄弟

### 4.1 RBD 块存储（最常用）

必须掌握的核心功能：

**快照（Snapshot）**：
```bash
rbd snap create rbd_pool/my_image@snap1
rbd snap list rbd_pool/my_image
rbd snap rollback rbd_pool/my_image@snap1  # 回滚
rbd snap rm rbd_pool/my_image@snap1
```

**克隆（Clone）**：
```bash
# 先给快照加保护，不然删不了
rbd snap protect rbd_pool/my_image@snap1

# 从快照克隆出新镜像
rbd clone rbd_pool/my_image@snap1 rbd_pool/cloned_image

# 分层 flatten（把依赖的快照数据拷过来，独立了）
rbd flatten rbd_pool/cloned_image
```

**精简配置**：
- 默认就是精简配置，写多少占多少
- 注意监控实际使用率，不要超售太多

### 4.2 RGW 对象存储

核心概念：
- **Bucket（桶）**：对象的容器，相当于文件夹
- **Object（对象）**：实际的数据，key-value 形式
- S3 兼容 API，几乎所有 S3 工具都能直接用

### 4.3 CephFS 文件存储

- 完整 POSIX 兼容，像 NFS 一样用
- MDS 管理元数据，支持动态子树分区
- 适合日志收集、媒体素材库这种场景
