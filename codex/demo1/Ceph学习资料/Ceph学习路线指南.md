## 📚 第一阶段：前置知识（必须先补）

很多人一上来就装 Ceph，结果踩的全是基础不牢的坑。

| 知识模块 | 核心内容 | 重要程度 |
|---------|---------|---------|
| **Linux 系统基础** | 磁盘分区、LVM、文件系统(XFS)、mount 选项、systemd 服务管理、日志查看 | 🔥 必须精通 |
| **存储基础** | 块存储/对象存储/文件存储的区别、SAN vs NAS、RAID 原理、SSD vs HDD 特性 | 🔥 必须精通 |
| **网络基础** | TCP/IP、VLAN、网卡绑定(mode4)、MTU 巨帧、延迟和带宽的区别、TCP 调优 | 🔥 必须精通 |
| **容器/K8s** | Docker 基础、K8s PV/PVC/StorageClass 概念、CSI 驱动 | ⭐ 非常重要 |

> ❌ 跳过前置知识的后果：
> 你永远不知道为什么 OSD 又慢了，为什么集群重平衡业务就卡爆，为什么 RBD 挂载超时。

## 🌱 第二阶段：入门基础（1~2 周）

目标：能搭集群，能看懂状态，能做基本操作。

### 2.1 核心概念（第一天就要懂）

👉 **Ceph 的设计哲学**：
- 没有单点故障的分布式架构
- 所有数据均匀分散，不存在元数据瓶颈
- CRUSH 算法是灵魂（不是一致性哈希！）

👉 **五大核心组件（必须背下来）**：

| 组件 | 角色 | 最少数量 | 说人话 |
|------|------|---------|--------|
| **MON** | 集群元数据管理、集群地图、认证 | 3 台起（奇数） | 集群的"大脑"，存集群的身份证 |
| **OSD** | 实际存数据的、处理 IO、数据同步 | 至少 3 个 | 集群的"搬运工"，干活的都是它 |
| **MGR** | 集群监控、管理接口、插件 | 至少 2 个 | 集群的"仪表盘"，可视化全靠它 |
| **MDS** | CephFS 文件系统元数据管理 | 可选 | 只有用文件存储才需要 |
| **RGW** | S3 兼容的对象存储网关 | 可选 | 只有用对象存储才需要 |

### 2.2 第一次部署（别用一键脚本！）

**正确的入门姿势：**
1. 用 KVM/VMware 开 3 台虚拟机，每台 2C 8G，加 2 块空盘
2. 先手动用 `cephadm` 一步一步搭（别用 rook，别用 ansible）
3. 搞清楚每个配置文件在哪，日志在哪，服务怎么起的
4. 部署完能看到 `HEALTH_OK` 就算入门了

**部署完要会的基本操作：**
```bash
ceph -s          # 看集群状态
ceph osd tree    # 看 OSD 分布
ceph df          # 看空间使用
ceph osd df      # 看每个 OSD 的使用率
rbd create ...   # 创建块设备
rbd map ...      # 挂载块设备
```

### 2.3 入门阶段要回答的问题
✅ 一个文件写入 Ceph，大概经历了什么流程？
✅ 为什么 MON 必须是奇数台？
✅ 三副本是怎么存的，存在哪？
✅ `HEALTH_WARN` 和 `HEALTH_ERR` 有什么区别？

## ⚡ 第三阶段：进阶核心（2~4 周，重中之重）

**这部分是 Ceph 的真正核心，也是面试必问，生产必用，排障必会的内容。**

### 3.1 CRUSH 算法（灵魂中的灵魂）

👉 **不是选学，是必须搞懂！**

你要搞懂的内容：
- CRUSH 和一致性哈希有什么区别？为什么 Ceph 不用一致性哈希？
- 什么是故障域？主机、机架、机房、地区层级怎么设计？
- CRUSH Map 怎么写，怎么导出，怎么修改，怎么生效？
- 权重是什么意思？为什么大硬盘权重要调高？
- 什么是 CRUSH Rule？副本规则和纠删码规则怎么写？

> 💡 懂了 CRUSH，你才算真正理解 Ceph。
> 90% 的容量不均衡、数据分布不合理问题，本质都是 CRUSH 没配好。

### 3.2 PG 原理（核心中的核心）

👉 **PG = Placement Group，Ceph 数据管理的基本单位**

必须搞懂：
- PG 到底是什么？为什么需要 PG，不能直接存对象？
- PG 数量怎么计算？公式是什么？太多太少有什么问题？
- PG 的状态机：`active+clean` / `degraded` / `remapped` / `peering` / `inconsistent` 分别是什么意思？
- Peering 过程是做什么的？为什么 Peering 慢会影响业务？
- PG 数量怎么调？什么时候需要调？

> 🚨 生产环境 80% 的故障，最终都会体现在 PG 状态异常上。
> 不懂 PG 状态，出了问题你连谷歌都不知道搜什么。

### 3.3 存储三兄弟

根据你的业务场景侧重学习，不用三个都精通：

| 存储类型 | 适用场景 | 核心知识点 |
|---------|---------|-----------|
| **RBD 块存储** | K8s 持久卷、虚拟机磁盘 | ⭐ 最重要：快照、克隆、精简配置、镜像、discard、分层 |
| **RGW 对象存储** | 兼容 S3 的对象存储 | 桶策略、生命周期、多区域同步、版本控制 |
| **CephFS 文件存储** | 分布式文件系统 | MDS 主备、动态子树分区、配额、快照 |

### 3.4 副本 vs 纠删码

必须搞懂的选型逻辑：
- 三副本：空间利用率 33%，性能好，恢复快，适用性能敏感场景
- 纠删码(EC)：空间利用率高（比如 4+2 是 67%），性能差，恢复慢，适用冷数据归档
- 纠删码的 K+M 怎么选？为什么不推荐 2+1？
- 纠删码可以做 RBD 吗？可以，但需要最新版本 + 开销不小

## 🔬 第四阶段：高级原理（高手分水岭，1~2 个月）

到了这一步，你和那些只会 `ceph -s` 的运维就拉开差距了。

### 4.1 BlueStore 深度原理（现在的标准）

👉 别再研究 FileStore 了，新版本全是 BlueStore。

必须搞懂：
- BlueStore 的架构：Block、DB、WAL 三个分区分别是干什么的？
- 为什么 DB 和 WAL 要放 SSD 上？性能提升多少？
- BlueStore 的 IO 写入路径：用户写请求进来，到底经历了什么？
- RocksDB 在 BlueStore 里扮演什么角色？
- FileStore 和 BlueStore 有什么区别？为什么 BlueStore 性能好？

### 4.2 一致性与可靠性

- 三副本的写入流程：主 OSD 收到写请求后，怎么同步到两个从 OSD？什么时候返回成功？
- 怎么保证副本之间的数据一致？
- Scrub 和 Deep Scrub 是做什么的？为什么必须定期跑？
- 静默数据损坏是什么？Ceph 怎么检测和修复？

### 4.3 缓存与性能

- 为什么不推荐用 Cache Tier 了？（90% 的场景都是负优化）
- 怎么正确用 SSD 加速：分离 DB/WAL 才是正道
- 内存怎么配？一个 OSD 需要多少内存？为什么 OSD 内存占用会那么高？

## 🚀 第五阶段：实战与运维（生产必备）

> Ceph 的运维经验是踩坑踩出来的，不是看书看出来的。

### 5.1 集群规划（部署前就要想清楚）

**硬件规划：**
- CPU：一个 OSD 至少配 1 个物理核心，纠删码要更多
- 内存：一个 OSD 至少 4G，推荐 8G
- 磁盘：HDD 做数据盘，SSD/NVMe 做 DB/WAL，不要用 RAID！
- 网络：必须分离公网（业务）和集群网（数据同步），10G 起步，25G 标配，100G 高性能

**容量规划：**
- 实际可用空间 = 总裸容量 ÷ 副本数
- 预留 20% 空闲空间给重平衡和恢复
- 单 OSD 使用率不要超过 80%

### 5.2 日常运维

这些操作必须练到肌肉记忆：
- 怎么安全地添加/移除 OSD？
- 怎么换坏盘？正确的换盘流程是什么？
- 怎么滚动升级 Ceph 版本？
- 怎么调整 CRUSH Map 而不影响业务？
- 怎么做 RBD 快照和备份？
- 怎么监控 Ceph？哪些指标是必须报警的？

### 5.3 故障排查（最重要的能力）

常见故障排查流程：
1. 集群不健康？先看 `ceph -s`，看是 PG 问题还是 OSD 问题
2. OSD 挂了？先看日志，是磁盘坏了还是网络问题还是内存 OOM
3. PG 不一致？先 `ceph health detail`，然后 `rados list-inconsistent-pg`
4. 性能慢？先看是单 OSD 慢还是全集群慢，再看磁盘 IO，再看网络

> 💡 最好的学习方法：自己搭个测试集群，故意搞破坏然后修。
> 杀 OSD、拔磁盘、断网络、删 PG，多搞几次，你就成高手了。

### 5.4 云原生集成

现在 90% 的新 Ceph 集群都是跑在 K8s 上用 Rook 管理：
- Rook 是什么？它和原生 Ceph 是什么关系？
- 怎么用 Rook 部署 Ceph？
- RBD CSI 驱动怎么用？快照、扩容、克隆怎么搞？
- Rook 运维的坑点有哪些？

## ⚠️ 新手最容易踩的坑

1. **不分公网和集群网** → 重平衡的时候把业务带宽打满，全站卡爆
2. **PG 数量算错** → 要么太少数据不均衡，要么太多耗光内存
3. **给 OSD 做 RAID5** → 脱裤子放屁，Ceph 本身就有副本，RAID 还拖慢性能
4. **硬盘满了才扩容** → Ceph 满了会直接锁写，提前规划
5. **一次加太多 OSD** → 触发全集群重平衡，带宽占满业务卡成狗
6. **不跑 Deep Scrub** → 磁盘静默坏了都不知道，等坏第二个盘数据就没了
7. **用 Cache Tier** → 90% 的场景都是负优化，还容易出一致性问题

## 💡 学习建议

1. **别光看，多动手**：Ceph 是玩出来的，不是看出来的。搭个集群，搞破坏，修，重复 10 次你就懂了。
2. **别追求大而全**：如果你只用来做 K8s RBD，那 RGW 和 CephFS 可以先放一放。
3. **别用太老的版本**：至少从 Pacific 开始，推荐 Quincy 或者 Reef，新版本坑少很多。
4. **生产环境别瞎试**：所有操作先在测试集群验证过了再上生产。
