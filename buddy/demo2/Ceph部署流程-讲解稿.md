# Ceph 分布式存储部署流程 — 讲解稿

> 面向听众：需要了解 Ceph 部署全流程的同事
> 用时：约 15~20 分钟（纯讲），现场演示另算

---

## 一、开场：Ceph 是什么（1 分钟）

一句话：**Ceph 是一个分布式存储系统，把多台服务器的硬盘整合成一个统一的存储池。**

它提供三种存储方式：
- **RBD** — 块存储，像虚拟硬盘，挂到 Linux 上用
- **CephFS** — 文件存储，像 NFS 共享目录
- **S3 对象存储** — 像阿里云 OSS，通过 API 读写

---

## 二、为什么要用 3 台机器（2 分钟）

Ceph 默认是 **3 副本模式**：你写入一份数据，它自动复制到三台不同的机器上。

- **3 台 = 最小标准方案** — 每一台同时跑 MON（监控）+ MGR（管理）+ OSD（存储），不浪费机器
- **为什么必须 3 台？** — 因为 Ceph 的 MON 节点用 Paxos 算法做决策，需要过半同意才能运行。3 台坏 1 台还能正常，2 台坏 1 台就瘫痪了，所以最少 3 台

硬件上配置很轻量：

| 资源 | 推荐值 | 原因 |
|------|--------|------|
| CPU | 4 核 | 每个 OSD 需要 1 核，MON/MGR 再占一些 |
| 内存 | 6GB+ | Ubuntu 24.04 + 容器运行时 + Ceph 进程 |
| 系统盘 | 50GB | 操作系统 + Ceph 镜像 |
| **数据盘** | **1 块裸盘** | **最关键**——必须是独立磁盘，不能是系统盘分区 |

## 三、部署流程总览（1 分钟）

整个搭建分为 **6 步**，但核心逻辑只有三件事：

```
1. 装系统 → 2. 配 SSH → 3. 跑 bootstrap（拉起集群骨架）
                                     ↓
                              4. 加其他节点 → 5. 加 OSD 盘 → 6. 验证
```

> 只有**第 1 步**需要在所有三台机器上重复执行，后续操作都在 ceph1 上完成。

---

## 四、分步讲解（10 分钟）

### 第 1 步：系统初始化

三台机器各自装完 Ubuntu 24.04 后，需要统一做几件事：

- **设主机名 + hosts 解析** — 让三台机器能互相通过名字找到对方
- **关 swap** — Ceph 强制要求，否则影响稳定性
- **装 podman** — Ceph 新版把所有的守护进程都跑在容器里，需要容器运行时
- **配 chrony 时间同步** — MON 节点之间对时间误差要求小于 50 毫秒，否则集群会报警
- **关防火墙** — 测试环境省事，生产环境需要逐端口开放

> **踩坑提示**：Ubuntu 24.04 默认禁止 root 通过 SSH 密码登录，需要提前改 `/etc/ssh/sshd_config` 配置，否则后续步骤会卡住。

### 第 2 步：SSH 免密配置

ceph1 要能无密码 SSH 登录到三台机器，因为后续所有操作都是通过 ceph1 远程下发到其他节点的。

```bash
# 生成密钥，分发到三台
ssh-keygen
ssh-copy-id root@ceph1  # 包括自己
ssh-copy-id root@ceph2
ssh-copy-id root@ceph3
```

### 第 3 步：bootstrap — 拉起骨架

这是整个流程里最关键的一行命令：

```bash
cephadm bootstrap --mon-ip 192.168.12.176
```

这条命令做了什么：
1. 从 `quay.io` 拉取 Ceph 容器镜像（约 1.3GB）
2. 在 ceph1 上启动第一个 **MON**（监控）和 **MGR**（管理）进程
3. 生成集群的配置文件、SSH 密钥、admin 账号密码
4. 启用 Dashboard（Web 管理界面）

**执行完后终端会输出 Dashboard 地址和密码，一定要保存。**

此时集群只有一个节点，执行 `ceph -s` 会看到 `HEALTH_WARN`（提示没有 OSD），这是正常的。

> **点睛之处**：cephadm 会自动为集群生成一套独立的 SSH 密钥（`/etc/ceph/ceph.pub`），和前面步骤 2 配置的 root 密钥是两套。需要把这套密钥也分发到其他节点，否则后续加节点时会提示 Permission denied。

### 第 4 步：加入其余节点

```bash
# 把 ceph2、ceph3 注册到集群
ceph orch host add ceph2 192.168.12.90
ceph orch host add ceph3 192.168.12.169

# 扩展 MON 到 3 个，实现高可用
ceph orch apply mon 3
```

`ceph orch apply mon 3` 的意思是让 cephadm 自动在集群里部署总共 3 个 MON 进程，它会自动分配到 ceph2 和 ceph3 上。等 1~2 分钟后，3 个 MON 组成法定人数，坏一台也不影响集群运行。

### 第 5 步：添加数据盘（OSD）

每台虚拟机的 `/dev/sdb` 是额外挂载的数据盘。OSD 就是 Ceph 用来存数据的进程，一个 OSD 对应一个物理磁盘。

```bash
# 查看哪些盘可用
ceph orch device ls

# 添加所有可用盘
ceph orch apply osd --all-available-devices
```

> **踩坑提示**：bootstrap 节点的盘有时不会被自动识别，可以用精确命令单独添加：`ceph orch daemon add osd ceph1:/dev/sdb`

### 第 6 步：验证

```bash
ceph -s
```

看到 `HEALTH_OK`、3 MON、3 OSD 全部 up，就说明集群搭建完成。

## 五、容量说明（2 分钟）

| 指标 | 数值 |
|------|------|
| 每台数据盘 | 60GB |
| 总原始容量 | 3 × 60GB = 180GB |
| 副本数 | 3（存三份） |
| **实际可用** | **180 ÷ 3 ≈ 60GB** |

Ceph 是存三份，不是 180GB 都能用。这不是"浪费"，而是用 2/3 的容量换可靠性——任意一台机器坏了，数据不会丢，集群照常读写。

---

## 六、后续能做什么（2 分钟）

集群搭好后，可以做的事情：

- **RBD 块存储** — 创建虚拟硬盘，挂载到 Linux 上使用（最常见）
- **CephFS 文件共享** — 多台机器共享同一个目录，类似 NFS
- **Dashboard 图形管理** — 浏览器访问 `https://192.168.12.176:8443`，查看容量和健康状态
- **客户端挂载** — 其他机器装个 `ceph-common`，通过网络远程使用 Ceph 存储

## 七、常见翻车点总结（1 分钟）

| 翻车点 | 原因 | 一句话解决 |
|--------|------|-----------|
| SSH 连不上 | Ubuntu 24.04 默认禁 root 密码登录 | 改 `sshd_config` 重启 |
| ceph orch host add 报错 | 没分发 `ceph.pub` | `ssh-copy-id -f -i /etc/ceph/ceph.pub` |
| bootstrap 节点的盘没识别 | 残留分区信息 | `wipefs` 清盘后手动指定 |
| 镜像拉不下来 | 墙 | 配加速器或提前 `podman pull` |
| MON_CLOCK_SKEW | 时间不同步 | 检查 `chronyc tracking` |

## 八、Q&A 准备

**问：为什么不用 ceph-deploy？**
答：ceph-deploy 已经被官方废弃了，cephadm 是当前唯一被推荐的部署方式，所有守护进程跑在容器里，管理起来更方便。

**问：Ubuntu 24.04 为什么不用 Reef？**
答：Reef 的 Ubuntu 包针对的是 22.04，在 24.04 上有 Boost 库依赖冲突。Squid 是第一个完整支持 24.04 的版本。

**问：生产环境需要变什么？**
答：① 不关防火墙，开放固定端口 ② 数据盘用 SSD ③ 网络建议万兆 ④ 角色分离（MON 独立 3 台，OSD 多台）
