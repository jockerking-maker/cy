# 医院药品管理系统 — 核心流程介绍

> 用于答辩陈述、演示讲解和回答「系统核心流程是什么」类问题。  
> 内容基于项目实际代码与数据库设计整理。

---

## 一、30 秒总述（答辩开场用）

> 本系统的核心是一条**药品进销存业务闭环**：
>
> **采购员**创建采购订单 → **审核员**审核 → **库管员**验收入库并更新库存 → 临床领用时**出库扣减库存** → 系统自动**库存预警** → 必要时对问题药品**锁定**禁止出库。
>
> 全流程采用**三权分立**（采购、审核、入库分角色），关键操作在**数据库事务**中完成，保证账实一致。

## 二、系统核心业务全景图

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        医院药品管理核心业务闭环                            │
└─────────────────────────────────────────────────────────────────────────┘

  【基础数据】                    【主业务流程】                 【安全与监控】
  药品信息 ──┐
  供应商   ──┼──→ 创建采购单 ──→ 采购审核 ──→ 入库登记 ──→ 库存台账
  仓库信息 ──┘        ↑              │            │            │
                      │              │            │            ├──→ 出库扣减
                 低库存预警 ─────────┘            │            │
                 触发补货采购                      │            ├──→ 库存盘点
                                                  │            │
                                                  └──→ 自动预警 ├──→ 药品锁定
                                                               └──→ 预警处理
```

**一句话概括：**

> 药品从「买进来」到「存起来」到「发出去」，中间有审核把关、库存实时更新、异常自动预警。

## 三、角色与流程分工（三权分立）

| 流程环节 | 执行角色 | 角色编码 | 核心职责 |
|----------|----------|----------|----------|
| 维护药品/供应商 | 采购员、管理员 | PURCHASER / ADMIN | 基础数据维护 |
| 创建采购订单 | 采购员 | PURCHASER | 发起采购申请 |
| 审核采购订单 | 审核员 | AUDITOR | 合规审批 |
| 入库/出库/盘点 | 库管员 | WAREHOUSE | 库存实物流转 |
| 预警处理/锁定 | 库管员 | WAREHOUSE | 风险管控 |
| 用户/角色管理 | 管理员 | ADMIN | 系统运维 |

**答辩话术：**

> 采购、审核、入库不是同一个人操作，形成相互制约，避免一人包办导致的合规风险。

## 四、流程一：用户登录与权限加载

### 4.1 流程说明

用户访问系统 → 输入账号密码 → 后端验证 → 签发 JWT → 前端保存 Token → 按角色加载菜单 → 进入工作台。

### 4.2 流程图

```
用户打开登录页
    │
    ▼
输入用户名 + 密码 + 验证码
    │
    ▼
POST /api/user/login
    │
    ├── 用户不存在 / 密码错误 / 账号禁用 ──→ 返回错误提示
    │
    └── 验证通过
            │
            ├── 生成 JWT Token（有效期 24 小时）
            ├── 查询用户角色（ADMIN / PURCHASER / AUDITOR / WAREHOUSE）
            ├── 查询角色对应菜单权限
            └── 返回 Token + 用户信息 + 角色 + 菜单
                    │
                    ▼
            前端存入 localStorage
                    │
                    ▼
            路由守卫校验 Token，按角色渲染侧边栏菜单
                    │
                    ▼
            跳转 Dashboard 首页
```

### 4.3 关键规则

| 项目 | 说明 |
|------|------|
| 认证方式 | JWT，请求头 `Authorization: Bearer {token}` |
| 密码校验 | BCrypt 加密比对（兼容旧 MD5 数据） |
| 前端权限 | `router/index.js` 路由守卫 + `requiredRoles` |
| 后端权限 | `AuthInterceptor` 拦截 + `@RequireRole` 注解 |
| 未登录 | 返回 401，前端跳转登录页 |
| 无权限 | 返回 403，前端提示并跳转首页 |

### 4.4 涉及代码

| 环节 | 文件 |
|------|------|
| 登录页 | `drug-front/src/views/Login.vue` |
| 登录接口 | `SysUserController.java` → `SysUserServiceImpl.login()` |
| JWT 生成 | `JwtUtil.java` |

## 五、流程二：药品采购与审核（核心主流程 · 上段）

### 5.1 流程说明

采购员根据库存或临床需求创建采购订单 → 审核员审核 → 通过后订单变为「已审核」，库管员方可据此入库。

### 5.2 流程图

```
【采购员】
    │
    ▼
选择供应商 + 添加药品明细（药品、数量、单价）
    │
    ▼
系统自动计算订单总金额
    │
    ▼
提交采购订单（status = 0 待审核）
    │
    ├── 写入 purchase_order（主表）
    └── 写入 purchase_order_item（明细表）
            │
            ▼
【审核员】
    │
    ▼
查看待审核订单列表
    │
    ▼
审核操作（通过 / 驳回 + 填写审核意见）
    │
    ├── 写入 audit_record（审核记录，可追溯）
    │
    ├── 审核通过 → status = 1（已审核）
    │
    └── 审核驳回 → status = 4（已驳回）
            │
            ▼
【库管员】（下一流程：入库）
    仅「已审核」订单可关联入库
```

### 5.3 采购订单状态

| status | 含义 | 说明 |
|--------|------|------|
| 0 | 待审核 | 采购员刚创建 |
| 1 | 已审核 | 审核通过，可入库 |
| 2 | 已入库 | 关联入库完成（数据库设计） |
| 3 | 已取消 | 采购员/管理员作废 |
| 4 | 已驳回 | 审核不通过（代码实现） |

### 5.4 关键规则

- 采购单号唯一，重复则拒绝保存
- 主从表结构：一张订单对应多条明细
- 审核记录独立存储在 `audit_record`，含审核人、结果、意见、时间
- 审核接口仅 **ADMIN、AUDITOR** 可访问

### 5.5 涉及代码

| 环节 | 文件 |
|------|------|
| 采购页面 | `views/purchase/PurchaseOrderList.vue` |
| 审核页面 | `views/purchase/PurchaseAuditList.vue` |
| 创建/审核接口 | `PurchaseOrderController.java` |
| 审核记录 | `AuditRecordServiceImpl.java` |

### 5.6 答辩话术

> 采购流程体现了**三权分立**：采购员负责申请，审核员负责审批，库管员负责收货入库，职责分离、记录可追溯。

## 六、流程三：药品入库（核心主流程 · 中段）

### 6.1 流程说明

库管员对已审核采购单办理入库，填写批次、生产日期、效期、数量，系统在同一事务中写入入库记录并更新库存，随后自动检测是否触发预警。

### 6.2 流程图

```
【库管员】
    │
    ▼
选择关联采购订单（可选）、药品、入库仓库
    │
    ▼
填写入库信息
    ├── 批次号（可自动生成：PC + 年月日 + 序号）
    ├── 生产日期
    ├── 有效期
    ├── 入库数量
    └── 入库单价
    │
    ▼
后端校验（数量>0、效期合法、批次不为空等）
    │
    ▼
【@Transactional 事务开始】
    │
    ├── 1. 生成入库单号 RK + 时间戳
    ├── 2. 保存 drug_in 入库记录
    ├── 3. 更新 drug_stock 库存
    │       ├── 同药品+仓库+批次已存在 → 累加 stock_num
    │       └── 不存在 → 新建库存记录
    └── 4. 调用 checkAndCreateWarning() 检测预警
    │
    ▼
【事务提交】入库完成
```

### 6.3 库存更新规则

| 维度 | 说明 |
|------|------|
| 库存粒度 | 药品 ID + 仓库 ID + 批次号 |
| 同批次入库 | 在原有 `drug_stock` 记录上增加 `stock_num` |
| 新批次入库 | 插入新的 `drug_stock` 记录 |
| 批次号格式 | `PC` + `yyyyMMdd` + 3 位序号，如 `PC20260511001` |
| 入库单号格式 | `RK` + `yyyyMMddHHmmss` |

### 6.4 关键校验

- 入库数量必须 > 0
- 生产日期、过期日期不能为空
- 过期日期不能早于生产日期
- 过期日期不能早于当前日期

### 6.5 涉及代码

| 环节 | 文件 |
|------|------|
| 入库页面 | `views/inout/DrugInList.vue` |
| 入库接口 | `DrugInController.java` |
| 核心业务 | `DrugInServiceImpl.saveDrugIn()` |
| 批次号生成 | `DrugInServiceImpl.generateBatchNo()` |

### 6.6 答辩话术

> 入库是库存增加的**唯一正规入口**。我在 Service 层用 `@Transactional` 保证「写入库单」和「改库存」要么同时成功，要么同时回滚，避免账实不符。

## 七、流程四：药品出库（核心主流程 · 下段）

### 7.1 流程说明

库管员为科室领药、门诊发药等场景办理出库，系统先校验可用库存，再按**先进先出（FIFO）**原则扣减批次库存，并触发预警检测。

### 7.2 流程图

```
【库管员】
    │
    ▼
选择药品、仓库、出库类型（领用/销售/报损/调拨等）
    │
    ▼
填写出库数量、单价
    │
    ▼
【库存校验 checkStock()】
    │
    ├── 统计该药品在该仓库的未过期库存
    ├── 计算：可用库存 = Σ(stock_num - lock_num)
    │
    ├── 可用库存 ≤ 0 ──→ 拒绝（已全部锁定或无库存）
    ├── 可用库存 < 出库数量 ──→ 拒绝（库存不足）
    │
    └── 校验通过
            │
            ▼
【@Transactional 事务开始】
    │
    ├── 1. 生成出库单号 CK + 时间戳
    ├── 2. 保存 drug_out 出库记录
    ├── 3. 按 FIFO 扣减库存 updateStock()
    │       └── 按 expiry_date 升序，优先扣减快过期批次
    └── 4. 调用 checkAndCreateWarning() 检测预警
    │
    ▼
【事务提交】出库完成
```

### 7.3 可用库存计算公式

```
可用库存 = 总库存数量(stock_num) - 锁定数量(lock_num)

其中：
- 只统计未过期的批次
- 被锁定的数量不能用于出库
```

### 7.4 FIFO 出库策略

| 步骤 | 说明 |
|------|------|
| 1 | 查询该药品在该仓库所有未过期批次 |
| 2 | 按 `expiry_date` 升序排列（先到期先出） |
| 3 | 从最早批次开始扣减，不够则继续下一批次 |
| 4 | 扣减时跳过已被锁定的数量 |

**答辩话术：**

> 出库采用**先进先出**策略，优先消耗临近效期的批次，降低药品过期损耗，这符合医院药事管理的实际规范。

### 7.5 涉及代码

| 环节 | 文件 |
|------|------|
| 出库页面 | `views/inout/DrugOutList.vue` |
| 出库接口 | `DrugOutController.java` |
| 库存校验 | `DrugOutServiceImpl.checkStock()` |
| FIFO 扣减 | `DrugOutServiceImpl.updateStock()` |

## 八、流程五：库存预警（自动触发）

### 8.1 流程说明

系统在每次入库、出库导致库存变化后，自动检测是否触发预警条件，生成预警记录供库管员处理。

### 8.2 流程图

```
入库 / 出库完成，库存数量变化
    │
    ▼
StockWarningService.checkAndCreateWarning(drugId, warehouseId)
    │
    ├── 汇总该药品在该仓库的可用库存
    │
    ├── 可用库存 ≤ warning_num（最低预警值）
    │       └── 生成「低库存预警」（type=0）
    │
    ├── 可用库存 ≥ max_warning_num（最高预警值）
    │       └── 生成「库存积压预警」（type=1）
    │
    └── （另支持临期/过期/滞销预警扫描）
            │
            ▼
库管员在「预警中心」查看并处理
    │
    ▼
标记 handle_status = 已处理，填写处理备注
```

### 8.3 预警类型

| warning_type | 名称 | 触发条件 |
|--------------|------|----------|
| 0 | 低库存预警 | 可用库存 ≤ 最低预警值 |
| 1 | 库存积压预警 | 可用库存 ≥ 最高预警值 |
| 2 | 临期预警 | 效期临近（如 180 天内） |
| 3 | 过期预警 | 已超过有效期 |
| 4 | 滞销预警 | 长期无出入库活动 |

### 8.4 预警级别（低库存示例）

| warning_level | 级别 | 判定规则 |
|---------------|------|----------|
| 2 | 紧急 | 可用库存 = 0 |
| 1 | 重要 | 可用库存 < 最低预警值 × 50% |
| 0 | 一般 | 其他低库存情况 |

### 8.5 防重复机制

> 同一天、同一药品、同一仓库、同一预警类型、未处理的记录，不会重复创建。

### 8.6 涉及代码

| 环节 | 文件 |
|------|------|
| 预警页面 | `views/stock/WarningCenter.vue` |
| 自动检测 | `StockWarningServiceImpl.checkAndCreateWarning()` |
| 临期扫描 | `StockWarningServiceImpl.checkAndCreateNearExpiryWarning()` |
| 预警处理 | `StockWarningServiceImpl.handleWarning()` |

### 8.7 答辩话术

> 预警不是人工定时查，而是**嵌入出入库流程的事件驱动机制**，库存一变就自动检测，及时提醒补货或促销出库。

## 九、流程六：药品锁定与解锁

### 9.1 流程说明

当药品存在质量问题、待召回或暂停使用时，库管员锁定部分库存，使其不可出库，问题解决后再解锁。

### 9.2 流程图

```
【锁定】
库管员选择药品 + 仓库 + 批次 + 锁定数量 + 锁定原因
    │
    ▼
校验：锁定数量 ≤ 可用库存(stock_num - lock_num)
    │
    ▼
【事务】
    ├── 写入 drug_lock 锁定记录（单号 SD + 时间戳）
    └── drug_stock.lock_num += 锁定数量
            │
            ▼
该部分库存不可用于出库

【解锁】
库管员对锁定记录执行解锁
    │
    ▼
【事务】
    ├── 更新 drug_lock.status = 已解锁
    └── drug_stock.lock_num -= 解锁数量
```

### 9.3 与出库的关系

```
出库可用量 = stock_num - lock_num

锁定后：lock_num 增大 → 可用量减少 → 出库校验更严格
```

### 9.4 涉及代码

| 环节 | 文件 |
|------|------|
| 锁定接口 | `DrugLockController.java` |
| 锁定/解锁逻辑 | `DrugLockServiceImpl.java` |

## 十、流程七：库存盘点

### 10.1 流程说明

库管员定期核对系统库存与实际库存，发现差异后同步调整台账。

### 10.2 流程图

```
【创建盘点单】
选择仓库 → 系统自动加载该仓库全部库存作为账面数
    │
    ├── 写入 stock_check（status=0 盘点中）
    └── 写入 stock_check_item（system_num = 当前库存数）
            │
            ▼
【填写实盘】
库管员逐项填写 actual_num（实际数量）
    │
    ▼
系统计算 diff_num = actual_num - system_num
            │
            ▼
【完成盘点】
    │
    ├── 若 diff_num < 0（盘亏）→ 扣减 drug_stock
    ├── 若 diff_num > 0（盘盈）→ 增加 drug_stock
    └── stock_check.status = 1（已完成）
```

### 10.3 盘点单状态

| status | 含义 |
|--------|------|
| 0 | 盘点中 |
| 1 | 已完成 |
| 2 | 已取消 |

### 10.4 涉及代码

| 环节 | 文件 |
|------|------|
| 盘点页面 | `views/stock/StockCheckList.vue` |
| 盘点接口 | `StockCheckController.java` |

## 十一、完整主链路串联（答辩重点 · 建议配合演示）

### 11.1 端到端业务链

```
Step 1  【采购员 caigou】创建采购订单
           ↓
Step 2  【审核员 shenhe】审核通过
           ↓
Step 3  【库管员 kuguan】关联采购单办理入库
           ↓  drug_stock 库存增加
Step 4  【库管员 kuguan】办理出库（科室领用）
           ↓  drug_stock 库存减少（FIFO）
Step 5  系统自动检测 → 若低于预警值，生成低库存预警
           ↓
Step 6  【库管员】在预警中心查看并处理
           ↓
Step 7  【采购员】根据预警再次创建采购单 → 形成闭环
```

### 11.2 2 分钟口述稿（完整版）

> 老师，我介绍一下系统的核心流程。
>
> 首先是**采购审核流程**。采购员选择供应商和药品明细创建采购订单，订单初始为待审核状态；审核员审核通过后，订单变为已审核，审核意见写入审核记录表，全程可追溯。
>
> 第二是**入库流程**。库管员根据已审核订单办理入库，填写批次、生产日期和效期。系统在事务中完成三件事：保存入库记录、更新库存台账、触发预警检测。
>
> 第三是**出库流程**。出库前先校验可用库存，可用量等于总库存减锁定量；校验通过后按先进先出原则扣减批次库存，并再次触发预警。
>
> 第四是**预警与锁定**。库存低于下限或高于上限时自动生成预警；问题药品可锁定，锁定部分禁止出库。
>
> 这四条流程串联起来，构成了从采购到使用的完整闭环，也是本系统的核心价值。

## 十三、核心技术保障（流程层面的技术点）

| 技术点 | 在流程中的作用 | 答辩怎么说 |
|--------|----------------|------------|
| `@Transactional` | 入库/出库/锁定/盘点保证原子性 | 「一步失败全部回滚，保证账实一致」 |
| FIFO 算法 | 出库优先消耗近效期批次 | 「降低过期损耗，符合药事规范」 |
| 事件驱动预警 | 出入库后自动检测 | 「不是定时任务，而是库存变化即时触发」 |
| RBAC 权限 | 每个流程环节分角色执行 | 「三权分立，岗位权责清晰」 |
| JWT 认证 | 全流程接口安全访问 | 「除登录外所有接口需 Token」 |
| 主从表设计 | 采购订单头+明细 | 「符合数据库范式设计」 |

## 十四、常见问题快速回答

**Q：系统最核心的流程是什么？**

> 采购 → 审核 → 入库 → 出库 → 预警，这是主链路；锁定和盘点是辅助保障流程。

**Q：如何保证库存数据准确？**

> 三方面：① 入库/出库在同一事务中更新库存；② 出库前校验可用量；③ 定期盘点校正差异。

**Q：预警什么时候触发？**

> 入库或出库完成后，系统自动调用 `checkAndCreateWarning`，对比药品的最低/最高预警值。

**Q：为什么出库要用 FIFO？**

> 优先消耗近效期批次，减少药品过期浪费，是医院库存管理的通行做法。

**Q：采购审核驳回后怎么办？**

> 订单标记为已驳回，审核记录保留；采购员可修改后重新提交或作废订单。
