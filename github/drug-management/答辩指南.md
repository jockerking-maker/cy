## 三、项目陈述稿（8～10 分钟，建议按此顺序讲）

### 3.1 研究背景与意义（1 分钟）

**怎么说：**

> 医院药品管理涉及采购、入库、存储、出库等多个环节，流程复杂、数据量大、安全性要求高。传统 Excel 或纸质台账方式存在三个主要问题：
>
> 1. **效率低**：重复录入、人工对账耗时  
> 2. **易出错**：账实不符、批次效期难以追踪  
> 3. **难协同**：采购、审核、库管职责分离，信息传递不及时  
>
> 因此我设计了一套 B/S 架构的药品管理系统，用信息化手段提升药事管理规范化和精细化水平。

**关键词**：全生命周期管理、批次追溯、效期管理、三权分立

---

### 3.2 系统目标与角色设计（1 分钟）

**系统目标：**

- 实现药品信息、供应商、仓库等基础数据管理
- 实现采购申请 → 审核 → 入库 → 出库完整业务链
- 实现库存预警、药品锁定等安全管理
- 实现 RBAC 多角色权限控制
- 提供 Dashboard 数据可视化

**四类角色（必须背熟）：**

| 角色 | 编码 | 主要职责 |
|------|------|----------|
| 系统管理员 | ADMIN | 用户管理、角色权限、系统公告、全部功能 |
| 采购员 | PURCHASER | 药品信息、供应商、创建采购订单 |
| 采购审核员 | AUDITOR | 审核采购订单（通过/驳回） |
| 库管员 | WAREHOUSE | 入库、出库、库存、盘点、预警、锁定 |

**设计亮点一句话：**

> 采购、审核、入库实现了**三权分立**，不同岗位只能操作职责范围内的功能，保证流程合规。

### 3.5 核心业务流程（2 分钟，重点！）

#### 流程 1：采购全流程

```
采购员创建订单 → 状态：待审核
    ↓
审核员审核（通过/驳回，写入 audit_record）
    ↓
审核通过 → 状态：已审核
    ↓
库管员按采购单入库 → 更新库存 → 状态：已入库
```

#### 流程 2：入库（强调事务）

> 入库时在一个**数据库事务**中完成：
> 1. 写入 `drug_in` 入库记录  
> 2. 更新或新建 `drug_stock` 库存  
> 3. 更新采购订单状态  
> 4. 触发库存预警检测  
>
> 任一步失败则全部回滚，保证数据一致性。

#### 流程 3：出库（强调校验）

> 出库前校验：**出库数量 ≤ 可用库存（quantity - lock_num）**  
> 校验通过后扣减库存，并触发预警检测。

#### 流程 4：库存预警

> 每次入库/出库后自动检测：
> - 库存 < `warning_num` → 低库存预警  
> - 库存 > `max_warning_num` → 积压预警  
> - 近效期/过期 → 效期预警  
>
> 预警级别按差距比例划分：紧急 / 重要 / 一般。

#### 流程 5：药品锁定

> 质量问题药品可锁定部分库存，`lock_num` 增加后不可用于出库，从源头阻断问题药品流通。

**建议画一张简图或指着 PPT 流程图讲，老师很爱问这块。**

### 3.6 数据库设计（1 分钟）

**怎么说：**

> 数据库名 `hospital_drug`，共设计 **19 张表**（论文中核心表约 15 张，实际还包含操作日志等辅助表），主要实体包括：

| 类别 | 核心表 |
|------|--------|
| 基础数据 | drug_info、supplier_info、warehouse_info |
| 采购 | purchase_order、purchase_order_item、audit_record |
| 出入库 | drug_in、drug_out |
| 库存 | drug_stock、drug_lock、stock_check、stock_check_item |
| 预警 | stock_warning |
| 权限 | sys_user、sys_role、sys_menu、sys_role_menu |
| 其他 | system_notice、sys_operation_log |

**E-R 关系一句话：**

> 供应商 1:N 采购订单，采购订单 1:N 明细，药品与仓库通过 drug_stock 形成多对多（含批次维度）。

**索引设计（可能被问）：**

- `drug_info.drug_code` 唯一索引
- `drug_stock(drug_id, warehouse_id)` 联合索引
- `purchase_order(status)` 状态索引

### 3.7 安全设计（1 分钟）

**怎么说：**

> 安全方面做了四层防护：

1. **身份认证**：JWT Token，24 小时过期，请求头 `Authorization: Bearer xxx`
2. **权限控制**：RBAC 模型，前端路由守卫 + 后端 `@RequireRole` + `AuthInterceptor`
3. **密码安全**：BCrypt 加密存储（兼容旧 MD5 数据迁移）
4. **攻击防护**：MyBatis-Plus 参数化查询防 SQL 注入；Vue 自动转义防 XSS

**前后端双重权限控制（加分点）：**

- 前端：无权限菜单不显示，路由跳转拦截
- 后端：接口级角色校验，未登录 401，无权限 403

## 五、高频提问与参考答案

### 5.1 选题与背景类

**Q1：为什么选择这个题目？有什么实际意义？**

> 医院药品管理是药事服务的核心环节，直接关系用药安全和运营效率。传统人工管理难以满足批次追溯、效期管控和多岗位协同的需求。本系统将采购、审核、入库、出库、预警串联起来，减少人工差错，提升信息化水平，对中小型医院有实际应用价值。

**Q2：你的系统和市面上 HIS/药房系统有什么区别？**

> 本系统是**面向药品进销存管理的专项系统**，聚焦采购—库存—预警闭环，而不是完整的 HIS。优势在于轻量、易部署、角色清晰；不足是未对接临床处方、医保结算等模块。后续可通过 RESTful API 与 HIS 集成。

**Q3：目标用户是谁？**

> 医院药事管理部门：采购员、审核员、库管员、系统管理员。不涉及患者端和医生开方。

---

### 5.2 技术选型类

**Q4：为什么用 Spring Boot 而不是 SSM 或其他框架？**

> Spring Boot 基于 Spring，提供自动配置和 Starter 依赖，减少 XML 配置，开发效率高；内嵌 Tomcat，部署简单；生态成熟，便于集成 MyBatis-Plus、JWT 等。相比传统 SSM，更适合快速构建 RESTful 后端服务。

**Q5：为什么用 Vue 3 而不是 React 或 jQuery？**

> Vue 3 学习曲线平缓，Composition API 便于组织逻辑；Element Plus 提供成熟的表格、表单组件，适合管理后台；与 Axios、Vue Router、Pinia 集成方便。jQuery 不适合复杂 SPA，React 生态更重，对本项目而言 Vue 性价比更高。

**Q6：为什么用 MyBatis-Plus 而不是 JPA/Hibernate？**

> 药品管理有大量条件查询、分页和复杂 SQL，MyBatis-Plus 在保留 MyBatis 灵活性的同时提供通用 CRUD 和 Lambda 条件构造器，减少样板代码，更适合本项目的 CRUD + 业务 SQL 混合场景。

**Q7：为什么前后端分离？**

> 前后端职责分离，可并行开发；前端可独立部署和缓存静态资源；后端 API 可被 Web、移动端复用；降低耦合，便于维护和扩展。

**Q8：为什么选 MySQL？**

> MySQL 开源、成熟、社区活跃；InnoDB 支持事务和外键，适合库存一致性要求；8.0 支持窗口函数、JSON 等，满足统计需求；与 Spring Boot 集成简单。

---

### 5.3 架构与设计类

**Q9：介绍一下系统架构。**

> 五层架构：表现层 Vue3、接口层 Spring MVC、业务层 Service、数据访问层 MyBatis-Plus、存储层 MySQL。前后端通过 RESTful API 通信，JSON 格式，统一响应 `{ code, message, data }`。

**Q10：RBAC 是怎么实现的？**

> 用户关联角色（sys_user.role_id），角色关联菜单（sys_role_menu）。  
> - **前端**：路由 meta 配置 requiredRoles，beforeEach 守卫校验  
> - **后端**：`@RequireRole` 注解 + `AuthInterceptor` 拦截 `/api` 请求，校验 JWT 和角色  
> 双重校验，防止仅改前端绕过权限。

**Q11：JWT 认证流程是怎样的？**

> 1. 用户登录，后端验证用户名密码  
> 2. 通过后生成 JWT（含 userId、username、过期时间），返回前端  
> 3. 前端存 localStorage，Axios 拦截器每次请求带 `Authorization: Bearer token`  
> 4. `AuthInterceptor` 解析 JWT，获取 userId，校验用户状态和角色  
> 5. Token 过期返回 401，前端跳转登录页  

**Q12：密码怎么加密存储的？**

> 新用户使用 **BCrypt** 单向哈希，自带盐值，抗彩虹表攻击。系统还兼容旧 MD5 加盐数据，登录验证时可识别并支持升级。论文中写的是 MD5，实际实现已升级为 BCrypt，这是答辩后可以说明的改进点。

**Q13：数据库设计遵循什么范式？**

> 主要遵循第三范式，减少冗余。例如采购订单与明细分表。部分场景适当反范式，如库存表保留批次信息、出入库记录冗余操作人姓名，以空间换查询性能。

**Q14：入库和出库如何保证数据一致性？**

> 使用 `@Transactional` 声明式事务。入库：写 drug_in + 更新 drug_stock + 更新订单状态 + 预警检测，同一事务。出库：校验可用库存 → 写 drug_out → 扣减库存 → 预警检测。任一步异常则全部回滚。

**Q15：库存是怎么计算的？**

> `drug_stock` 表按 **药品 + 仓库 + 批次** 维度存储 quantity 和 lock_num。  
> - 可用库存 = quantity - lock_num  
> - 入库增加 quantity  
> - 出库减少 quantity  
> - 锁定增加 lock_num  

**Q16：采购订单主从表怎么设计？**

> `purchase_order` 存订单头（供应商、总金额、状态）；`purchase_order_item` 存明细（药品、数量、单价）。一对多关系，创建订单时事务内同时插入主表和从表。

**Q17：预警是如何触发的？**

> 在入库、出库 Service 完成库存变更后，调用 `StockWarningService` 检测：对比 warning_num / max_warning_num，检查效期，满足条件则写入 stock_warning。属于**事件驱动**而非定时轮询（也可补充：可扩展定时任务做全量扫描）。

### 5.4 功能与业务类

**Q18：采购审核驳回后怎么办？**

> 订单保持待审核状态，审核意见写入 audit_record。采购员可修改后重新提交或取消订单，审核记录可追溯。

**Q19：出库时库存不足怎么处理？**

> 后端校验 `出库数量 > 可用库存` 时抛出业务异常，事务回滚，前端提示「库存不足」，不允许出库。

**Q20：药品锁定有什么作用？**

> 对质量问题或待召回药品锁定部分数量，lock_num 增加后这部分不可出库，避免问题药品流入临床，是安全保障措施。

**Q21：库存盘点的流程？**

> 创建盘点单，系统加载仓库库存作为账面数；库管员填实际数量，系统算差异；完成盘点时按差异调整 drug_stock，在事务中执行。

**Q22：Dashboard 展示哪些数据？**

> 药品总数、预警数、待审核采购单数、库存总金额；药品分类环形图；预警柱状图；近效期药品列表；系统公告时间轴。数据来自 DashboardService 聚合查询。

**Q23：系统支持哪些出库类型？**

> 科室领用、门诊领药、住院领药、销售、报损、调拨等，记录在 drug_out.out_type 字段。

### 5.5 安全与测试类

**Q24：如何防止 SQL 注入？**

> MyBatis-Plus 使用预编译参数绑定，用户输入作为参数而非 SQL 拼接，从机制上避免 SQL 注入。测试中输入 `' OR '1'='1` 等无效。

**Q25：如何防止 XSS？**

> Vue 模板默认转义输出；Element Plus 组件渲染文本而非原始 HTML；后端对关键字段做校验。

**Q26：你做了哪些测试？**

> 功能测试 15 项（登录、CRUD、采购审核、出入库、预警、权限等）；性能测试（Postman 10 并发）；安全测试（认证、越权、注入、XSS）；兼容性测试（四大浏览器）。详见论文第六章。

**Q27：性能测试结果如何？**

> 10 并发下，登录约 85ms，列表查询约 42ms，入库约 98ms，均满足「核心接口 < 1 秒、页面 < 3 秒」的非功能需求。

### 5.6 难点、创新与不足类

**Q28：开发中遇到的最大困难是什么？怎么解决的？**

> 可任选其一回答（选你真实经历）：
>
> **方案 A（库存一致性）**：  
> 入库同时要更新库存、订单状态并触发预警，一开始出现数据不一致。后来用 `@Transactional` 把多步操作放在同一事务，并加强出库前可用库存校验。
>
> **方案 B（权限控制）**：  
> 仅前端控制可被绕过。后来增加后端拦截器和 `@RequireRole`，实现前后端双重校验。
>
> **方案 C（前后端联调）**：  
> 跨域问题通过 Vite 代理和 CorsConfig 解决；统一 Result 响应格式便于前端处理。

**Q29：你的创新点是什么？**

> 1. 采购—审核—入库**三权分立**的业务流程设计  
> 2. 入库/出库后**自动触发**多维度库存预警（低库存、积压、临期、过期）  
> 3. **药品锁定**机制保障用药安全  
> 4. Dashboard **可视化**辅助管理决策  
> 5. 前后端分离 + RBAC 的完整权限体系  

（创新点不宜夸大，强调「业务闭环 + 安全 + 预警」即可）

**Q30：系统有什么不足？**

> 1. 单体架构，大规模扩展性有限  
> 2. 无实时消息推送（WebSocket）  
> 3. 未对接国家药品追溯码  
> 4. 缺少移动端  
> 5. 智能分析（需求预测、补货推荐）较弱  

（与论文 7.2 一致，主动说比被老师问出来更好）

**Q31：如果继续改进，你会先做哪一项？**

> 优先考虑 **WebSocket 预警推送** 或 **对接药品追溯码**，因为与医院实际需求最贴近，且能在现有架构上扩展。

### 5.7 论文与实现细节类（容易被追问）

**Q32：论文里写 MD5 加密，代码里是 BCrypt，以哪个为准？**

> 论文撰写时方案为 MD5 加盐；后续实现中升级为 BCrypt，安全性更高。PasswordUtil 保留对旧 MD5 数据的兼容。答辩以**实际代码 BCrypt** 为准，论文可注明「实现中已升级」。

**Q33：论文里写简单 Token，代码里是 JWT？**

> 同理，实现阶段采用了标准 JWT（jjwt），支持过期时间和签名验证，比简单字符串 Token 更安全。这是实现上的改进。

**Q34：为什么用 Java 17？**

> Spring Boot 3.x 要求 Java 17+，且 17 是 LTS 版本，性能和新特性（如 Records、Pattern Matching）更好。

**Q35：@RequireRole 注解是怎么工作的？**

> 自定义注解标注在 Controller 方法或类上，AuthInterceptor 在 preHandle 中通过反射读取注解，获取当前用户角色，与要求角色比对，不匹配则返回 403。

**Q36：软删除是怎么做的？**

> 部分表有 `is_deleted` 字段，MyBatis-Plus 可配置逻辑删除，查询时自动过滤已删除记录。

**Q37：批次号怎么生成？**

> 格式 `PC + 年月日 + 序号`，通过接口 `/api/drug/in/next-batch-no` 获取，保证唯一性。

**Q38：效期怎么计算？**

> 根据生产日期 + 药品 shelf_life（保质期月数）自动计算 expiry_date，前端展示时近效期、过期用不同颜色标签标识。

## 六、可能追问的「刁钻问题」及应对

| 问题 | 应对策略 |
|------|----------|
| 并发入库会不会超卖？ | 说明当前为课程设计规模；生产环境可加乐观锁/悲观锁（version 字段或 SELECT FOR UPDATE） |
| 为什么不用 Redis？ | 当前数据量和并发不大，MySQL 足够；Redis 可用于缓存热点药品、Session，是后续优化方向 |
| 为什么不用 Spring Security 全套？ | 项目采用轻量方案：Interceptor + JWT + 注解，满足 RBAC 需求，避免 Security 配置复杂度 |
| 数据库连接池用的什么？ | Spring Boot 默认 HikariCP |
| 事务隔离级别？ | 默认 READ COMMITTED（InnoDB），库存场景可考虑 REPEATABLE READ |
| 日志怎么记录？ | sys_operation_log 表 + 可扩展 AOP 记录操作日志 |
| 接口文档有没有？ | 可答 RESTful 设计见论文表 4-5；可补充 Swagger 是后续改进 |
| 单元测试写了没？ | 论文以功能/集成测试为主；单元测试可列为不足 |
